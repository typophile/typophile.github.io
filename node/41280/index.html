Well, kerning is just another feature, so you can just write the kerning (& other features) in a text editor, then import it into FontLab. Using this technique, you import the features using the OpenType panel, not the Metrics Window.

If you have already written the other features, write them off, add the kerning in the proper syntax, & (re)import the whole thing.

example of “kerning as a feature”:

feature kern { # Kerning

Latin

pos parenleft quoteleft -10;
pos parenleft quotedblleft -10;
pos parenleft ayn 10;
pos parenleft C.smcp -20;
pos parenleft five 20;

. . . .

} kern;

2008-01-24 22:21:14 • charles ellertson

FontLab Studio is also able to link to external .FEA files in the Adobe FDK syntax.

Produce a text file that looks like following:

feature kern { pos A quoteright -111; pos A comma 19; pos A period 19; pos A colon 19; pos A semicolon 19; pos A T -83; pos A U -42; pos A V -120; } kern;

and save the file as kerning.fea in the Features folder inside of your Studio 5 user folder (see the manual for specific locations).

In the lower-right part of the OpenType panel, place a line such as

include(kerning.fea);

When you generate an OpenType font, it’ll have the OpenType kerning as specified there. This way, you LINK the kerning to the VFB file, not IMPORT. To import, you need to open the .fea features file from the OpenType panel.

You can also import the kerning in the AFM file using the Metrics Window in FontLab Studio.

Of course, both OpenType kerning and AFM kerning can be imported into the font using FontLab Studio’s Python scripting language in a fully automated manner.

It should me made clear that,

  1. importing an AFM file through the Metrics Window, and
  2. adding the ‘kern’ feature in the OpenType panel do not have the same results.

The first will add the kerning data to the glyphs, while the second won’t. This means that if you use 2) you won’t see any kerning or be able to edit the kerning in the Metrics Window. In addition, if the Preferences are set to “Generate the OpenType ‘kern’ feature if it is undefined or outdated”, the code added in 2) will be blown away and probably replaced by an empty ‘kern’ feature.

Well, yes and no. If you import the kerning data via a kern feature, and remember NOT to update the kerning feature when you generate an .otf font, then if in FontLab you open the .otf font you’ve just generated (rather than the .vfb where you did the work), the kerning will be “in the font” – e.g., you can write off an .afm, etc. You could, if you wished, then reset the classes & features & start afresh, use the metrics window, etc.

Of course, if you are going to go this far, it is simpler to do as Miguel suggests and simply generate an .afm, add the kerning with the correct syntax for an .afm, and read it back in.

BTW, If the kerning is from Quark, as I remember you have to multiply all the values by eight.
