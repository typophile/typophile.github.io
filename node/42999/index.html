A very good place to get under the skin of CSS (not really to learn it, but to understand it on a deeper level) is the CSS Zen Garden. Click on some of the designs on the right and see the power of CSS—the XHTML markup is precisely the same for all pages, only the CSS is changed. Once you feel good and comfortable with XHTML and CSS, I’d suggest learning some basic JavaScript and PHP. These two languages are the two most widely used ways of creating dynamic content and database interaction, at least in *NIX-based server environments. In Windows-based environments, ASP is probably more common than PHP, but I’d learn PHP anyway, since it, unlike ASP, is platform-independent.

I think Asvetic has identified a key point on the web design learning curve here, and one that colleagues and I have had problems with. This is how I see many individuals grasp of HTML/CSS develops (I'm writing quickly between jobs so hope it makes sense): One discovers early on that writing HTML *does* determine both layout and appearance of a document - because browsers have inbuilt stylesheets, invisible to the user, thus ways of rendering it. This confuses right from the start by conflicting with the "correct" model of HTML being purely structural. Re Oisin above, the buttons, images and text would show up in the right place on screen with no use of CSS at all. CSS is then found to be a convenient way of globally specifying fonts and colours, maybe margins, but not much more - it's a fairly superficial layer which *adds* to the appearance of plain HTML. People often find that more complete uses of CSS fail to render as they expect, so abandon further experimentation. For example, trying to lay out complex forms. Or when trying to put together an HTML email. Tables-for-layout work better for both, so are seen as a reliable method, and the individual's understanding of the HTML+CSS model stalls - this is why they are sometimes seen as alternatives. I think use of 'reset' stylesheets can help with this by eliminating the inbuilt browser styling.

Actually, the hardest part of a web designers job is to realize that the user controls much more than the designer does (in the true spirit of HTML). The user decides the size of the page, always. I did teach HTML, for about 10 years. I always started with a course in HTML, hard coding with Notepad (or the Mac text editor of your choice). CSS was a second course, and then, after the students knew code, they were introduced to Dreamweaver and FrontPage (this was back when the latter had a trace of relevance.) Within the HTML portion, we spent considerable time building pages that were dynamic ... in other words, they would be workable at any size then current (640x480 was the minimum at that time, and later 800x600 was the minimum). The max was what the monitors at any time would take. I remember changing the screen background image so that it contained boxes of 640x480 and 800x600 as well as full screen so that the students would reduce their browser windows to those dimensions to see how the page would look to folk with older computers. The other big concern I remember at that time was that pages had to work correctly in both the current IE and Netscape (today it would be IE and Firefox). The only other browser at that time with much share was Opera, but such a small usage that it could be safely ignored. One thing I didn't do at that time, that I would now, is to insist on validated code. Anyone wanting to submit invalid code would have to supply me with an essay explaining why the end results justified the invalid code.

"If YOU had to teach someone how to do web design/development, what would your teaching outline look like? Where would you start and what would come next?" Good question. Not an entire course, but specifically the process to build an HTML page: 1) Get the content. All of it. 2) How to mark it up with HTML in a semantic manner and talk about issues like SEO and Accessibility. 3) How to layout out the page structure via CSS (and adding the necessary additional HTML as needed). 4) How to now bring your visual layout (photoshop mockup) into the mix. 5) Polishing it all up adding the images, typography, color, to the CSS file. 6) Now let's talk about why it looks like crap in IE and how to fix it. ;o) "What would invalid code be considered, is that the same as deprecated code?" Invalid code is code that doesn't validate: http://validator.w3.org/ The closest analogy I can think of is pre-press flight checking. Deprecated code is usually something that has been replaced with something 'newer and better' though the 'better' part can often be debated. ;o)
