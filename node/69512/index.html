Hello Tim,

FontLab can’t decompile TT instructions from generated TTF fonts. If you do TrueType hinting in FontLab you destroy the existing (binary) instructions. If you want to see something like the picture above you have to apply (PostScript) autohinting and convert this to TT instructions.

The kind of TT instructions is indicated as icon in the font window. If you see a blue ‘T’ in the lower left corner you have binary instructions which can’t be edited. A yellow ‘T’ indicates visible and editable TT instructions.

2010-04-16 15:57:06 • Goran Soderstrom

Let me add to what Eigi wrote that there are probably two reasons why existing TT instructions cannot be imported and edited in FLS:

  1. FontLab Studio supports only a subset of the TT instruction set
  2. It's not trivial (or possible at all) to decompile TT instruction programs and find out which value, e.g. in the cvt table, stands for what, because it's all just numbers ;)

You can however decompile TT instructions using ttx on a font, but then you would see the raw instruction code and not a visual representation like in FLS.

2010-04-16 22:58:37 • dberlow

The visual TrueType instructions in FontLab Studio (or VTT) are kind of like source code, while the binary TrueType instructions in a TTF font are, well, like binary code. You can decompile the binary into assembly code (e.g. using TTX) but it’s not at all trivial to decompile into the visual source code, especially since the source code in FontLab Studio provides a particular set of functions that does not necessarily represent all of situations that you can code using the TrueType assembly.

Unlike, say, the syntax that is used in OpenType Layout tables (GPOS, GSUB) – which is a very simple language and is actually more like data than like programming code – the TrueType assembly language is a real programming language in which, for example, you have GOTO-like jumps.

FontForge has a TrueType instruction debugger that lets you step through the native TrueType instructions and watch how the outline gets modified by the TT instruction code.

2010-04-20 03:56:14 • .00

The TT instructions you are looking at is just like a stack crawl.

The numbers you see are the point numbers, cvts, or other numeric values that are pushed onto the stack first then what follows are the instructions that use (or pop) these values. The MDAP is what is commonly called an Anchor and the MIRP is what could be called a link.

If the MDAP is the first instruction that pops values it will take the first value (which is probably a 1 for ‘round the value’). It also pops the next value which will be the outline point number. It does not push anything back on the stack so the third value is now at the top of the stack.

The TrueType specification (originally from Apple) explains what each instructions does and what the Boolean values are for.

2010-04-21 02:24:22 • Michel Boyer

First and most important — as it has been pointed out e.g. here on Typophile — the two main Apple patents on TrueType hinting expired six months ago.

The U.S. patent 5155805 (“Method and apparatus for moving control points in displaying digital typeface on raster output devices”) was filed on May 8, 1989 and issued on October 13, 1992, and it expired on October 13, 2009.

The U.S. patent 5159668 (same title) was filed on May 8, 1989 and issued on October 27, 1992, and it expired on October 27, 2009.

The U.S. patent 5325479 was filed on May 28, 1992 and issued on June 28, 1994. It is likely to expire on May 28, 2012. However, this was the second filing of the same patent as 5155805, and the first one was granted only after the second filing was done. As far as I understand, the term of expiration of the very first (oldest) filing counts.

So it can be assumed that all the relevant patents have expired now.

Second, patents typically describe a “method and apparatus for something”. The patents in question did not describe a particular arrangement of software programming instructions that would be the “method and apparatus”. The patented “method and apparatus” was, simply speaking, the TrueType instructions programming language (or aspects of how it works). In other words, the interpreter, not any particular code that should be interpreted, was patented.

Writing instructions for the interpreter did not infringe the patents. Writing an interpreter that would execute those instructions did.

But as I said, those patents expired now, so the discussion about myths is kind of moot :)
