> If the pdf creator stores the double, tripple > assigments too, it should be possible to > search these glyphs in a proper way. Andreas, you’re talking about introducing more flexibility in the domain where we, font developers, have influence (the way glyphs are named or encoded in the font) but on the other hand you want other domains (applications) to worry about making sure that everything still works fine for the user. In other words, you want to buy yourself some freedom at the expense of application developers. But this will not work. Font developers are a smaller community and therefore can work out certain standards of behavior better. Application developers are a very large group and their efforts are rather uncoordinated. For example, you say "if the pdf creator". But there are 200 different PDF creators on the market, some relying on partial implementations of the standard (e.g. the Quartz engine in Mac OS X), some relying on implementation of older PDF versions, some relying on the PostScript-to-PDF conversion etc. At least in half of the scenarios, OpenType PS fonts are converted to Type 1 fonts (so cmap, GSUB etc. are all removed and only glyph names travel into the file) and the additional Unicode stream is not stored. Therefore, the PDF really only has a visual glyph representation of the text, there is no metadata. Such PDFs will ALWAYS be better searchable if the each glyph has a unique glyphname and just one Unicode codepoint. You should not think of PDF being only dependant on what Acrobat is doing. The problem is less and less with Acrobat because Acrobat is getting smarter and smarter. But there are many other tools for searching inside of PDFs, for example plugins for Spotlight, Google Desktop Search or Microsoft search etc., and there is no guarantee that they will ever be as smart as Acrobat. The point in font development, especially with OpenType, is to try to make the fonts work as good as possible in *various* systems and applications. This often includes making some compromises or doing redundant steps to make sure that certain information is present in three different places. I wish it were different but *we* don't have any way to change it. If you don't like the way the Adobe FDK for OpenType writes the cmap table, you can always change it with a tool like TTX or you can use FontLab Studio 5 to build your fonts. Adam

2006-09-08 12:05:42 • andreas
Andreas, you can only decompile and recompile a specific table, e.g. ttx -t cmap yourfont.otf will produce a file "yourfont.ttx" with only the cmap table. You can edit it, rename to "yourfont2.ttx" and merge into your .otf using ttx -m yourfont.otf yourfont2.ttx The result will be a font yourfont2.otf which will contain the new cmap and the rest of the tables will be identical to those in the original yourfont.otf. In fact, in most cases, you can use one and the same cmap-only .ttx file for the entire family (so you dump and edit once, and then merge it into all fonts in your family that have the same glyphset). In FontLab Studio 5.0.2, standard.nam (Standard Table) has double-mappings for mu, Delta and Omega, while the AGLFN 1.6 (new) strict table only has the new AGLFN 1.6-compatible assignments. It is explained in the 5.0.2 announcement: http://fontlab.com/The-News/Announcements/FontLab-Studio-5.0.2/ as well as in the .nam files themselves. Adam A.
