Once we’re working at these sizes, (below 12pt) the rasterizer, OS, and formats used become critical. You’re working on a project for the web, so presumably the end user can use any modern OS and browser. What’s of concern is how you are rendering the image. Are you using TrueType OTF or CSS/OTF? What rasterizer are you using? A Win, OSX, or something custom?

Another comment: regarding your comment, “..gets very jagged.” A hand-hinted TTF/OTF on a Win system, would have the gasp table set to show B+W rendering at sizes below 12pt. This is because Anti-aliasing at these sizes tends to cause more problems than it solves. So the characters are rendered B+W (like a bitmap) so that it remains legible. This does result in a stair step affect, but it’s a compromise that needs to be made, if you’re hinting these small sizes.

Another note, although there are more than a couple of dozen hand-hinted quality TTF/OTF fonts, the stylistic differences at these sizes all but disappear, as David mentioned,. The truth is, there just aren’t enough pixels to go around. This means that the differences between Arial, Helvetica, and even Frutiger or Optima, all disappear at these small sizes, when you have to make a legible glyph, and have no room for stylistic differences.

All the rest is sane, just be a bit careful using “pt”, because e.g. “12 pt” can contain anywhere from ((LESSTHAN))12 pixels per em to 32 or more per em depending on OS and display device. Since that makes it possible for a 9 pt to be 20 pixels, or more, and 9 being a smaller size than 12, we always have to think in pixels, not points, like, I believe a gasp table.
