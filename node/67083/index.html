CFF fonts do not use composites natively. PS fonts use subroutinisation for outline optimisation. This is a font generation-time option in FL and other tools. So yes, if you are building FontLab projects using composites, you need to set the option to decompose during font generation and, if desired, use subroutinisation. TTF fonts do use composites natively, i.e. the TT glyf table uses composites in much the same way as they are defined within FontLab. Composites inherit hinting from base glyphs, so they are a useful way not only to reduce the size of the font but also to reduce the hinting workload. [I recommend against scaling, strethcing, flipping or otherwise manipulating composites -- other than shifting x,y position -- as possible in FontLab, though, as this seems to cause rendering problems.] In recent years, I've started to quite regularly see fonts including glyphs made up of overlapping components, e.g. a /Ccedilla/ glyph in which a /C/ base component and /cedilla/ mark component are used, rather than these being decomposed and the outlines merged. I guess this indicates that the makers of these fonts have sufficient faith in the current state of TT rasterisers, and consider poential problems associated with overlapping contours to be obsolete concerns.
