–

If you sell font licenses for the web usage, please please make them Cufon friendly. No self-respecting pirate will ever care to convert obscured Javascript outlines back into mangled OTFs, when it is far easier to find the originals on torrent sites. But having a Cufon option will make the fonts far more attractive and useful to the web designers.

If you are set on a service business model with recurrent charges, consider offering Cufon .js the same way you currently offer @font-face files. In the end this is not about paying less or more, it is about having a website look the same on all platforms. Something that is currently impossible to do with just the @font-face.

Hey pank, I’m a Cufon fan, too. But why not @font-face for Mac and Flash Replacement for everyone else?

And I think there is much, much less concern these days for consistent appearance from device to device, browser to browser. Half the time, it seems like nobody tests further than Safari on Mac. Anyway, embracing and delighting in the differences is going to make for greater happiness. Learn to love ugly - I’m amazed at how forgiving people seem to be about it when it comes to fonts.

I am mulling over writing a replacement rendering engine for Cufon that would outsource the rasterization to the web server. In other words when it would come to actually drawing text on the canvas, it would instead talk to a web server, have it draw that text into a .png instead and then fetch and display the resulting image. Basically a form of FLIR that works as a drop-in replacement for Cufon.

If I am to take the idea a bit further, this could actually be wrapped in a service run by the foundries themselves. And under the very same recurrent billing model used for @font-face licensing.

2011-06-08 21:35:48 • quadibloc

… the good news (from my perspective) is that Linden Hill, from the same source, works.

Of course, Cufon has serious limitations. CSS has to be used for leading - that isn’t really a limitation, except that CSS is slightly more awkward to use with it. But the big limitation is that you can’t have it substitute multiple fonts, giving each font certain tags - although there’s a workaround involving a fancy JavaScript utility that turns various scripts on and off.

The fact that the technique it uses means that people have to “view source” to copy text off the page… that will annoy people, and start some on their first step to becoming hackers. For some content, of course, it’s largely irrelevant.

Of course, for an open-source font, embedding is not required, and so @font-face probably is the right way to go for most purposes.
