  1. Does hinting affect printing? Hinting is not “kind of output” specific — it is resolution and typeface design specific. If one composes type at 72 pt on a 96 dot per inch screen (96 pixels per vertical em), pending typeface design issues, hinting does not effect the type. If one composes 10 pt for a 144 dpi printer, (20 pixels per vertical em), hinting likely is needed to save the type regardless of typeface design issues. Typeface design for less hinting means making fonts more regular in every imaginable way, (until one reaches Chicago;) And for more hinting, typeface design should travel towards Snell Roundhand or Augsburger Initials.

  2. How is the type rendered in the preview window in FontLab? Adam, Yuri? How is T1 rendered? How is TT rendered, please? Obviously from above, the larger one renders type in FL, the more accurate, the smaller one renders type in FL, the less accurate. And, the less regular the design, the larger one renders type in FL, the more accurate, the smaller one renders type in FL, the less accurate. Then, depending on how faithful FL is to the OS rendering of your choice, you have an answer.

  3. If I export a font before applying any hinting is it up to the engines of individual programs to cope with rendering the type on screen? All screen rendering depends first on the outlines, and then on hints if they are present. The outlines are either scaled and rendered, or scaled/hinted and rendered. Scaling in this context means shrinking from, e.g. a 1000 unit em to a 25 unit em to transform the outline (1000) to the bitmap ( 6 pt 300 dpi). In this scaling, each and every contour point in the outline must find a coordinate in the new space. So, again, the more regular the type is on the em, the more regular it’ll stay when it is scaled to a lower resolution em. Hints are nice if you want irregular type to remain irregular.

Then, when you project this out into OS and application space, all hell breaks loose unless the OS native to the font format handles it, in which case some hell breaks loose because, though the outlines may be regular or irregular, the hints are treated irregularly if used at all.

  1. What are the essentials for a reasonable laser proof? Exporting fonts/printing direct from FontLab, PostScript/emulated etc. It’s best to generate the final format you intend to supply and use it on the OS(s) and in the expected application(s) for output from the devices where you expect people to use it.

Cheers!

Most of the original questions have already been answered, but I have a few things to add.

First, not only does hinting affect printing, but bad hinting can deform letters when printing to a laser printer. Run the AFDKO autohinter before doing your test prints and the prints will look a hell of a lot better.

I have also found that using Fontlab to generate proofs only works down to nine or ten points. Below that the shapes get mangled, so test small sizes from Indesign.

3) How is the type rendered in the preview window in FontLab?

Depends on which platform you use … If I remember correctly, on Windows FL uses an (the?) Adobe rasterizer for PostScript Outline hinting preview (you can choose between Black/White and Greyscale), and for TrueType outlines the Windows rasterizer in Black/White, Greyscale or ClearType mode.

On Mac OS X it’s different, obviously the Microsoft rasterizer can’t be used, and that’s why the TT preview on Mac is so bad ;)

How trustworthy is this representation?

As trustworthy as you’re certain about which rasterizer will be used for displaying your fonts finally. In the end reliable hinting comes down to educated guessing and experience, I suppose:)

“What do you mean by regular on the em?”

Fonts for print usually have slight variations in otherwise common values, like alignments and stem thicknesses, to create the optical effects one sees in print as correct. These slight variations however, introduce rounding ‘errors’ at lower resolutions and one sees these on screens as incorrect. One prominent example is the O overshoot vs. H, which, if you look closely on your screen at this font, does not exist, though O overshoot does exist in the outlines of the font you are using. Within the overshoots, the O,C,J,G,S,Q &,& Ø may all have slightly different alignments and weights of the baseline round hairline(s) for print due to variety in shape and proximity of features, width of character, and contrast of baseline hairline, to create the optical effects one sees in print as correct. A more regular design might have one common overshoot, or in a squarer design, none and thus be even more regular on the em. And, the hairlines can be simplified down to a couple of values or even a single value, all to create the optical effects one sees on the screen as correct.

Hope that helps some.

Cheers!
