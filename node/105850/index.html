In theory this would be possible as long as your font is installed at the system level on the relevant computer and is set to be the default font (where that is possible) in the applications you are using. As soon as you change to a different font the correct character for that code point will be displayed, so the behaviour you are hoping to see is dependent on how you set up the computer(s) on which the font is installed.

In general this is very bad practice for designing fonts and as soon as you introduce the font to different environments over which you have little or no control you will start to see unexpected and unpredictable results. If the character you are creating adheres to a Unicode code point then it should be assigned to that code point in order to display consistently, regardless of which application you are using. If you are creating Chinese characters then you will find fewer fonts (depending on the computer) that support it but it will be universally (and correctly) recognised.

As far as I can tell, Private Character Editor maps its characters to code points in the Private Use Area. You could set up Fontlab Studio look and behave a bit more like PCE by toggling “Unicode ranges” mode in the Font window, then choosing “E000 Private Use Area” from the dropdown. (You can also assign Unicode values manually in the Glyph Properties box.) I don’t know enough about PCE and its TTE files, Font Links, etc. to know if generating an ordinary font with PUA-encoded glyphs would yield the result you’re after, but at least your custom characters would not be sharing Unicode values with existing characters.

It’s been a while since I’ve had to deal with characters made with the Private Character Editor, but it was painful. This software used to be known as EUDCEDIT, where eudc stands for “end user defined characters,” so you might try googling by that name. Michael Kaplan has a [[http://blogs.msdn.com/b/michkap/archive/2010/01/22/9951970.aspx|2010 blog post]] that lists the two files EUDCEDIT creates when used under WinXP, eudc.tte and eudc.euf, and mentions the difficulty of restoring the latter. My recollection is that the practical way to “liberate” the bitmaps was by screen-grabs.

Personally, I’d be really wary of relying on a word-processor’s customized input routine tied to a custom version of a standard font that retains the orignal’s name. And while creating one’s own Chinese input system might be fun, for serious use I’d rather learn one that has evolved through decades of use. I like 倉頡 (cangjie), though I gather 五笔 (wubi) is similar. Having spent the time to learn cangjie, I can toggle that IME on with a key combination, press a few keys to get my character, and then toggle back to continue typing with regular keyboard input. MS Word will automatically find a font for my character, even if it is a rare form up in Unicode’s CJK Extension B (note that accessing Extension B requires adjusting the IME’s default settings). And if I want to share my thoughts with others I don’t have to print them out: anyone else can read my file with their system’s standard software.

Good luck! David
