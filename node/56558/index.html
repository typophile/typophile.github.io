2009-04-03 05:33:08 • cuttlefish

I’ve always assumed that OT fonts work on both Windows and Macintosh platforms, as indeed they seem to do. What’s going on?

Nothing special :-)

Both ‘cmap’ and ‘name’ tables have more than a single data entry. To discern between entries, each one is prefixed with salient details. Platform is important because the default character maps differ significantly between Mac and Windows. Additionally, traditionally, Mac char maps were 8 bits – so for different languages, it had different mappings (entirely similar to ye olde DOS/Windows code pages). So a font just has loads of mappings, each one prepended with a system and a language code.

Names is sort of the same story. What if you have an accented character in a name string – say, the designer’s name? Should you use Mac standard encoding, Windows standard encoding, or Unicode? The canonical solution was: put it all in. Even though the 21st century solution is to use Unicode for everything, Macs historically expect a Macintosh version.

I deliberately say “historically”, because I’m pretty much convinced with OSX it’s no longer an issue. Those in the know may correct and/or amend to this, but I think both cmaps and names are chosen purely by ‘best fit’ (transcending encoding and platform), instead of insisting on a Norwegian style name encoded in a Windows Norwegian 8 bit code page for a Norwegian OS.

So, what’s going on? “Better safe than sorry” applies–as well as “Standards are good–let’s have lots of them”. A not-too-bad-coded font may be useable on any modern system (Win XP, Vista, OSX, or one of the more advanced linux desktops), but a font with all bells and whistles in every table is usable on your old Mac SE.

You’ll have to ask someone who maintains the OpenType spec, or the MS Recommendations, about that. I’m just reporting what I know. The Adobe specs for glyphnames make it pretty clear that “.null” is not a legal name: http://www.adobe.com/devnet/opentype/archives/glyphnamelimits.html (see #4, Length and Character Set Limitations on Names).

But again, there are many, many fonts with glyphs named “.null” which work just fine in many different implementations.
