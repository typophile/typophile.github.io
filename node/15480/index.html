2005-10-04 21:49:18 • paul d hunt
Well moved, Paul. At a glance, it would appear that PS Type 1 is now dead in the water, BUT it's a chicken-and-egg situation; the OpenType format represents a far smaller range than what we're now used to, because the collection is still being added to ('standard' faces being coverted from the 'standard' format). At the university where I work the current debate is about which format type library we purchase next year on a pretty sizable site license; as I understand it, if we spend, say $30K, we get an OpenType library that is less than a quarter of the size of the corresponding PS Type 1 library - and the PS Type 1 library would only cost a third as much again at around $40K. In type education terms, we might not need all 32 instances of Adobe Warnock Pro OTF - but we really do need all the weights of the Univers family, so my colleagues are still keen on going with the established formats and libraries ('standard' faces in the 'standard' format). In these terms PS Type 1 still represents good value for money. However, continued sales of PS Type 1 libraries may subsequently act as a disincentive to people or foundries developing OpenType fonts, and that may mean it's taking longer to become the next established standard.

This means that apps written specifically for Avalon will not support Type 1 fonts. Apps that still use the older GDI engine will continue to support Type 1 fonts. That, at least, is my understanding of the current state of play. At Tiro, we now have a policy of not making Type 1 fonts even if clients request them, and have turned down work on this basis. The main reason for this is the lack of QA tools for testing Type 1 fonts, and I don't like to ship software that I can't see inside. Compare the validation, comparison and testing tools for TT and CFF OpenType freely available from Microsoft and Adobe, and font table editing tools like TTX, to the kind of blindness in which we had to work on Type 1 fonts, just trusting that tools like Fontographer and FontLab were getting everything right. We wasted too many hours of our lives trying to simply identify bugs in Type 1 fonts, let alone trying to fix them. The last time we shipped Type 1 fonts, one key output device in our client's print production workflow substituted incorrect characters, despite all other proofing devices handling the font correctly. How do you fix a problem like that? How can you assure the client that the font is made to spec when there are no validation tools to ensure conformance? Learning to interpret raw PS code didn't seem like a good use of a type designer's time. Type 1 fonts? I spit on this arsewipe, ill-documented format. Good riddance.

2005-10-05 16:50:37 • silas
OpenType fonts will work with current and older versions of Quark, but with some limitations. You only get the first 200 or so characters--equivalent to MacRoman or Windows Latin-1, so they will seem just like plain old PostScript Type 1 or TrueType fonts. You also get no OpenType feature support (ligatures, alternate figures, small caps, etc.). Kerning will mostly work, but there may be combinations left out. OT fonts usually use class-based kerning. In order for kerning to work in, say, Quark 6, the OS has to create a table of kerning pairs derived from the class-based kerning. There is a limit to how big such a table can be. Class-based kerning can be the equivalent of thousands or even tens of thousands of kerning pairs, much higher than the limit in many cases. So a subset is used. In most other respects, OT fonts behave the same as older font formats in Quark.

John, why not provide T1 fonts but with an explicit no-support warning? It seems unfair to shun the format for people who are stuck with it. Who is stuck with it? Some people have workflows that include software with 8-bit text encoding limitations, but almost all such software provides codepage support so can access more than 256 characters from a large font. I don't want to encourage 8-bit hack encodings for typographic features, e.g. the Adobe 'expert set' encoding, but even if this is what the user wants this is an encoding issue not a format issue. Even if a user is truly limited to 8-bit fonts, this still doesn't imply a limitation to the Type 1 font format. I would happily ship TrueType or CFF fonts with specific 8-bit character set support if a client really needed them. Are there applications in current use* -- or perhaps output devices? -- that specifically require the Type 1 font format? There may be, but I can't think of any offhand. * By this I mean both that they are being used and that there are good reasons why a later version of the software cannot be used, e.g. because it doesn't exist or because of compatibility with some other aspect of the workflow.
