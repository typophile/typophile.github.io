Chartwell: ligature activated graphing tools

March 7, 2011 • traviskochel

Chartwell is a family that explores the use of OpenType to interpret and visualize data. The font format is highly portable and can be used in any application that supports standard ligatures. The data also remains editable allowing for easy updates. More info and licensing at http://tktype.com/chartwell.php [img:sites/default/files/old-images/Chartwell-Pies_6623.png] [img:sites/default/files/old-images/Chartwell-Bars_3941.png] [img:sites/default/files/old-images/Chartwell-Lines_5137.png]

@Pomeranz Thanks for sharing. @dberlow Thanks for the thoughts. It's good to hear some critical feedback. Yes, in the examples, the output has been scaled up. This was just a design decision to keep the examples simple, but perhaps it would be worth noting in the examples to be completely transparent. I'm OK with these only working in OT savvy environments. I see these as more of a tool to help designers create charts in their native design files. As far as the scaling issue goes, it's pretty easy to change the pt size, just like any font. This definitely is not ready for the web. I've got some ideas on how to use JavaScript to do the work of OT, in browsers not that don't support ligatures. But I think rendering is an issue, even on a Mac. The glyphs have to line up perfectly, or else you end up with gaps between values. I really don't have the manpower, or technical know-how, to manually TT these, because of the sheer number of glyphs. As riccardo pointed out, these aren't limited to black and white. You can give it any color you like. Whatever color is applied to the number, the related ligature glyph will take on that color. The decision to go with b&w was again, purely a design/brand decision. Treating it much like a standard type specimen, I thought it was best to keep it simple and show that these hold their own, even without color.

2011-03-11 21:51:03 • rosaiani
Nick, I thought a little bit more about another way to solve your problem, and came up with this theory. I haven't tested it, and it has some kinks to work out, but it may be a good starting point? Assuming you've already got tabular arbitrary fractions. Also assumes a space before and after the fraction. You really only need to know how many digits are in each the numerator and denominator. To do this, you could just swap classes with classes, and count. Figure out how many numerator positions you'd like to support, and create a class for each, plus a base. So for example, consider the fraction. 2384/4890 You'd need to support four numerators and denominators. Create 5 classes and name them something like. num_base [zero.sup one.sup two.sup...] all classes have exactly the same glyphs num_one [zero.sup one.sup two.sup...] num_two [zero.sup one.sup two.sup...] num_three [zero.sup one.sup two.sup...] num_four [zero.sup one.sup two.sup...] Then swap classes to count. sub space @num_base' by @num_one; Identifies first digit sub @num_one @num_base' by @num_two; identifies second sub @num_two @num_base' by @num_three; identifies third sub @num_three @num_base' by @num_four; identifies fourth Perform similar task for the denominators, except we'll count backwards from right to left this time, because the context with the fraction is most important. So... sub @denom_base' space by denom_one; sub @num_base' @denom_one @denom_one space by denom_two; sub @num_base' @denom_two space by denom_three; sub @num_base' @denom_three space by denom_four; So in this example, we'd end up with space @num_one @num_two @num_three @num_four / @denom_four @denom_three @denom_two @denom_one Now we've got to center everything. It's probably easiest to just center everything around the numerator, but that may give you issues crashing into previous text. I'll let you figure out how to solve that problem, but I'm sure it's possible. So first make the bars, with their varying widths to cover your options. If you give the fraction a negative left sidebearing, you can center it under the numerator. Then give it a negative right side-bearing, this can be used to center the denominator. A basic swap, with exactly the same number of digits in num/denom, you could do something like this. Sub @num_four fraction' by frac_four; or Sub fraction' @denom_four by frac_four; However to be a little more flexible, supporting varying numbers of digits, create four different versions of the fraction_four. each one with a different right sidebearing, to center the denominator. Then... Sub @num_four fraction' @denom_three by four_frac_three; And cover all possibilities. Like I said, I haven't tested this, but maybe it will help? If it works, it could sove having to make ton of extra glyphs. The only glyph that would need alternates is the fraction bar.
