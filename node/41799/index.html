February 7, 2008 • Typography.Guru

I open InDesign CS2 and InDesign CS3 an choose the font Windings. In CS2 the right pointing arrow shows up as F8FF in the glyph palette. In CS3 it shows up at F0F0. Each programm only accepts one code (even with hex input or PopChar), but not the other.

What’s going on?

That’s not exactly a “Unicode” issue (as in standardisation). Both codepoints you stated are from the Private Use Area.

To my best knowledge, Windings is actually 8-bit “Symbol” encoded. I suspect, InDesign maps “symbol”-encoded fonts to the PUA internally – it might be, that CS2 and CS3 versions use different mappings (which they can. That’s the catch [and the benefit] of the PUA, it’s not standardised, codepoint meanings are left to the user/application).

That said, I have never used InDesign and might have no clue…

The thing is that Wingdings uses a “Symbol” (3.0) cmap rather than a “Unicode” (3.1) cmap, so applications and operating systems need to deal with them in a special way. Microsoft Windows maps the 256 Symbol codes to U+F000-U+F0FF. My guess is that the InDesign CS2 (and previous) method was not really compatible with what Windows does (which is not to say that it was wrong, it was just different) but Adobe at some point decided to match the Windows behavior.

Ralf,

in a sense, the 3.0 Symbol cmap behavior is similar to the way applications and the OS treat Type 1 fonts – they synthesize Unicode values from glyphnames. Different operating systems have different lists of “known” glyphnames. For example, Adobe applications and the Windows font driver “know” the glyphnames “Ncommaaccent”, “Ncedilla” and “uni0145”, and synthesize the codepoint U+0145 for each of them. But Mac OS X 10.4 and earlier does not understand the glyphname “Ncommaaccent” so it fails to synthesize a codepoint if it finds such a glyph – but it does know “Ncedilla” and “uni0145”.

Also, most OpenType fonts have two Unicode cmaps: 0.1 (“Apple” Unicode) and 3.1 (“Microsoft” Unicode). Windows always looks at the 3.1 cmap while Mac OS X looks at the 0.1 cmap first, and if it does not fine one, at the 3.1 cmap. Font creation software would normally make sure that both cmaps have the same contents – but it does not come naturally. Once can imagine fonts that (for whatever reasons) need to present different glyphs to Mac OS X and Windows, and then the font developer could make the cmaps different. In fact, I have made such fonts for a client.
