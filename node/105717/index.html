I think what he means is that you're trying to use OTL as a kludge for something which would be much better achieved through a keyboard layout which either maps the vowel + number sequences to precomposed glyphs or which maps numbers to combining accents. The problem with what you're doing is that á isn't a ligature of a + 2. By treating it as such, your text stream is still going to consist of a2 even if á appears on screen which is going to interfere with search algorithms, etc., and which means that type entered in your font isn't going to work when you switch to another font. If you prefer to enter pinyin characters this way, a keyboard layout is going to work with any font which contains the required glyphs, and isn't going to lead to problems with searching or displaying text. I cannot support too forcefully the notion that the font is the wrong place to address a matter better handled by input method: for example, if you want to type 2nd-tone "a" in MS Word, you could set its automated spell-checking to replace an "a" followed by a "2" with á. Alternatively, there used to be Word macros for this. In addition it is worth noting that Unicode officially uses carons (a.k.a. hačeks) rather than breves to indicate pinyin's 3rd tone. Thus Unicode's official description for ǚ (U+01DA) is "Latin small letter u with diaeresis and caron." I wish it were an independent character, just in case I wanted to use a breve, as some folks do, but it isn't. So if I wanted to put 3rd-tone ü in a font with a breve instead of a caron I'd probably still encode it at U+01DA. But in fact I've given in and use carons when adding tone-marked pinyin vowels to a font: Unicode long since decided Chinese was "ideographic" (over sinologists' preferred "logographic"), in comparison with which haček vs. breve is small potatoes.
