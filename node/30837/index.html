Do ya like it Bold, or BLD?

January 24, 2007 • Choz Cunningham

I have read that best practices involve making stylistic components of a name as short as possible.

But, off the top of your collective heads, what do you personally prefer? What about when there are multiple tags on the end? Is consistency important? Compare below, and offer other strategies, if you like:

Option No. 1 (short/consistent)– [Fontname] [Cutname] [Fontname] [Cutname] Bld [Fontname] [Cutname2] Cnd Bld Obl [Fontname] [Cutname2] Obl

Option No. 2 (long/unambigous) [Fontname] [Cutname] [Fontname] [Cutname] Bold [Fontname] [Cutname2] Condensed Bold Oblique [Fontname] [Cutname2] Oblique

Option No.3 (mixed) [Fontname] [Cutname] [Fontname] [Cutname] Bold [Fontname] [Cutname2] Condensed Bld Obl [Fontname] [Cutname2] Oblique

Note: This applies only to OpenType naming fields only. The old-style names seem to be a messheap where no clear method works well.

2007-01-24 13:08:29 • Choz Cunningham

The numeric one promoted by Linotype – page 85 of my second edition of Stop Stealing Sheep (where I’ve taken the quote below) has the matrix for figuring out the weights of Univers, especially when you consider the permutations and combinations now available (59 weights plus 4 monospaced typewriter fonts!), it’s a snap to figure out which one you’ve got.

The first digit stands for weight, i.e., 1 is ultralight, 2 thin, etc., and 9 is extra black. The second digit denotes width, i.e., 1 for compressed, 2 for condensed, 3 for basic, and 4 for extended. The third indicates upright roman (0) or italic (1). Not exactly intuitive, but effective once you get used to it.

If you don’t want to go that route (I like the fact that it’s language-independent, personally), then I’d vote for your Option #2 – I’m big on keeping things as unambiguous as possible.

First, my answer is #2.

But at more length….

I have read that best practices involve making stylistic components of a name as short as possible.

Says who? Why on earth would you use cryptic abbreviations except when necessary?

The rule of thumb at Adobe is to make stylistic components of a menu name as close to the way one would actually read/write it as possible, within the constraints of:

A) Maximum name lengths of different fields in the font

B) For a given field, across fonts within a family, we want to be consistent.

So for an example of (B), if for the Mac Classic menu name we are forced to abbreviate “Italic” to “It” for Expialidocious Extended Bold Italic, then we’ll do it for the Mac Classic menu names for all the italic faces in the Expialidocious family.

2007-01-28 05:14:23 • Choz Cunningham

My short answer is, if you’re using FontLab, use the naming process outlined by Adam Twardoch and all will be largely well. http://groups.msn.com/FontLab/tipsandtricks.msnw?action=get_message&mview=0&ID_Message=2843

Gruesome details on OpenType and TrueType name IDs:

So for TrueType, OS X will use the Mac name IDs 1 and 2 for apps using the fonts panel. If you don’t do anything special in FontLab, these will be set the same as their Windows equivalents, and you’ll be stuck back at the four-member family model. (But there’s another option, so keep on reading.)

Apps getting “Classic” menu names will use Mac name ID 18 if present. (FontLab’s “Mac Name” under “OpenType-specific names.")

Adobe apps use Win name IDs 16 and 17 on both platforms. We consider those the typographically savvy names. (In FontLab: OT Family Name and OT Style Name.)

Note that if name IDs 16 and 17 are not present, name IDs 1 and 2 get used instead. So if they’re the same, they can be omitted. (FontLab does this automatically.)

Note that in Adobe fonts, Mac name IDs 1 and 2 usually correspond to Win name IDs 16 and 17. However, 16 and 17 have a slightly longer length limit than 1 and 2 (I think it’s 63 or 64 characters, instead of 31 or 32 characters). FontLab can set this up automatically if you check the option under “Generating OpenType & TrueType” for “use the OpenType names as menu names on Macintosh.”
