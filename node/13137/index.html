I’m making a (grayscale) pixelfont of an existing outline/print font.

Although the bitmaps of the glyphs can only occupy integer pixel widths, I need to assign to each glyph the non-pixel-integer set width from the outline font, which will (assumedly) be additioned cumulatively by the typesetting application, causing an occasional extra pixel shift, in order to match linebreaks/the_right_rag with the equivalent setting in the “source” outline/print font.

I assume the only way to do this is by assigning width adjustments to each glyph via hinting. I’m wondering if anybody can offer help in how best to do that, and what to worry about.

I realized this morning that the issue simply is not an issue. I thought that a character in a pixelfont that has a non-pixel-integer width would cause subsequent glyphs to get misaligned… but they don’t! So it turns out hinting is not even necessary. BTW, I also noticed -more like was reminded of- something else: apps like Photoshop use only the pixel-integer values from the bitmaps/rendering (not the actual em values) to place glyphs, while Word for example does use the underlying “true” width value (resulting in occasional unsightly gaps, but accurate line ends/breaks).
