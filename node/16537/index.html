When multiple keystrokes are entered to represent an OpenType composite glyph that is referenced with a ‘sub’ feature, such as typing option-e e on OS X to get an eacute, are the individual unicode indexes from the keystrokes recorded in the document or the unicode index for the composite?

For instance, an option-e e is U+02CA and U+0065. If an OpenType feature does a sub of these keystrokes to U+00E9 is the U+00E9 saved in the document or the ‘e’ plus the acutecomb? What about a ccmp feature that doesn’t point to a specific Unicode index?

Thanks,

What you are talking about are so-called dead-key combinations. The first keystroke changes the state of the keyboard to “attention, now follows a special keystroke”, and then the latter genenrates a codepoint. It’s usually only one codepoint, the precomposed character.

Of course you can write your own keyboard driver that produces any keystrokes you need. On Windows, there is the free Microsoft Keyboard Layout Creator (MSKLC).

OpenType features such as ccmp work on glyphs. From the perspective of an OpenType Layout feature, it does not matter if a glyph has an Unicode codepoint associated with it or not.

Regards, Adam

My opinion is that dead key technology is becoming obsolete. It is really only useful for arriving at precomposed diacritic characters, so while it will get you to your C with a caron (U+010C), it will not get you any further. My recommendation for dealing with Unicode text entry is to avoid the dead key method and instead design keyboards to enter sequences of base plus combining marks as discreet characters. These can then be resolved using either character level mapping in a text engine or glyph level substition (ccmp) or postioning (mark and mkmk). This is the approach I’ve taken in the new polytonic Greek layout I am working on, since I need to be able to support non-standard combinations that might occur in early manuscripts.
