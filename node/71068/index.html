2010-06-05 17:53:52 • Arno Enslin
Arno, the second question wasn't anything to do with ‘use of unlicensed fonts’, it was about checking whether a font has been modified. Plenty of licenses permit modifying fonts so long as the derivative versions are not distributed. There's no easy way to check whether a font has been modified without having the original against which to do a diff. ____ Re. the kerning question: there are two kinds of kerning that a font may have: kern table pair kerning, and GPOS table kerning (which may involve classes or, indeed, even funky things like contextual kerning, or kerning implemented as single glyph adjustments instead of pairs). When you examine a font using the MS Font Properties Extension, the 'Features' tab will indicate how many ‘standard kern pairs’ a font contains; this is the number of kern pairs in the kern table. If a font is not reported as having any standard kern pairs, this does not mean that it has no kerning: it may have GPOS kerning (check the OpenType Layout feature tree in the same tab). Font editing or examining tools do different things when looking at fonts with GPOS kerning. Some will parse the GPOS kern feature and report the information back in terms of pairs, but there are various ways in which this can be misleading. For instance, a tool may or may not expand class kerning, and since GPOS kerning, like all OTL features, may be script and language specific what data is actually being looked at? As noted in the Calibri kerning thread, tools also do different things when looking at kern table kerning. The MS font properties extension and DTL OTMaster will give you accurate pair counts (as will Ascender's FontChef tool). Other tools, including TTX, may not, depending on the size of the table. Note, however, that the number of pairs in the kern table and even a list of those pairs won't necessarily reflect the actual kerning that you're going to get in different applications. Some applications can't handle very large numbers of kern pairs, and some have weird bugs based on things like 8-bit codepage contents (e.g. MS Word kerning only monotonic Greek diacritics, not polytonic).

2010-06-05 21:38:35 • Fontology
Fontology, A first thing to check to see if a font has been modified or not, is to check its version number against an original. On Mac OSX, this is as simple as Get Info -- for Windows, you need Microsoft's own Font Properties extension. If the version number is different, that is almost certainly the reason. If the version number is the same but the modification date/time and file size is different, the font might have been edited -- most foundries are careful to maintain accurate version numbering, precisely for this reason.
2010-06-05 22:18:39 • Theunis de Jong
Charles — If your designer swears that the fitting is different than she’s used to and you’re able to confirm that the font files are “original” and unmodified, then check to see whether InDesign’s Optical Metrics feature has been applied. This feature very subtly, but very definitely, mucks up the spacing of a font — ignoring and overriding the font’s metrics and kerning. I just had a conversation with a colleague about a circumstance where the first proofs of a magazine redesign using a custom typeface design, which had been meticulously spaced and kerned, came back looking unbelievably atrocious, and my friend thought he was going crazy and began to question everything he thought he knew about spacing and H&Js. It turns out someone had ignorantly applied Optical Metrics. Upon turning this off and reproofing, everyone agreed that the composition and design suddenly looked infinitely better and miraculously fixed. I’ve ranted at length about this “feature” before on this forum, so I won’t repeat myself here. But just have your designer check for this.
