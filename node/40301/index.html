You need to be aware that OT does not know about anything like lines or words. All that can be checked is whether a glyph’s context consists of certain other glyphs.

So, it is not possible to define line beginning/ending contexts. What is possible however is defining word beginning/ending with help of a trick: There is a word beginning/ending if no other letter (a b c …) precedes or follows it: http://typophile.com/node/19839 I think there were one or two more recent threads about this but I cannot find them.

Again, I cannot think of any reliable way to identify line boundaries in OT. The space character is not a reliable partner for these kinds of affairs.

At first I thought of a trick like this:    (1) ignore sub e' @letter; sub e' by e.wordend; All end-of-word e’s are turned into wordend e’s. This identifies word boundaries only by the fact that no letter follows. Forget about space.    (2a) ignore sub e.wordend' [space nbspace]; sub e.wordend' by e.lineend; If no space follows (= line end), e.wordend becomes e.lineend. But this is not a good context since a line break may very well be preceded by a space … So we could try adjust (2a) a little:    (2b) ignore sub e.wordend' @space @letter; sub e.wordend' by e.lineend; Where @space = [space nbspace] and possible other space glyphs. The application may include a line break character. If so, the ignore statement’s context is not matched, and e.wordend substituted by e.lineend. But this too may not be reliable because different applications behave differently as regards automatic line breaks. I found that in some applications contexts are broken if an automatic line break interferes (InDesign), while in other applications the automatic line break is treated as non-existing (Mellel). [Things may have changed. I tested this one or two years ago.] So you would not get the same result in all OT-savvy applications. And as said above, space it not really reliable since applications may use another character than space or nbspace, which would break your context definitions.

My example code is speculative – I have not actually tested it. And for the downsides mentioned I would suggest that you deal with word-end letters in calt as you do, but for line-end letters better do single substitution in aalt or an ssXX feature so designers would assign the feature to individual letters. As much as I like automatism, in this case it would do more harm than good.

P.S. And only in a third step (lookup) substitute e e' by e.alt.
