2009-09-24 15:15:46 • kosal
Is there any circumstance where one would NOT want to decompose and merge? For fonts with TrueType outlines, it's not necessary to decompose composites in the final fonts. It may actually be desireable to keep accented glyphs as components because that way you only have to hint the accent and base glyph once and can use additional hinting in the composite glyph e.g. to keep the base glyph and accent from touching. If you decompose, these stacked hints would break. To be on the safe side, don't use scaled or flipped components if you plan to keep composites in your fonts.

Ad 7. If you're asking about the caption of a glyph cell being bright red, it means that you have a mismatch between the glyphname and the Unicode codepoint. Regarding decomposing and merging: FontLab Studio doesn't automatically remove overlaps because it is not always a perfectly precise operation. Sometimes, when you have complex shapes that overlap, there may be some manual clean up work required. However, there is a setting in Preferences to automatically decompose all glyphs when the font is generated as OpenType PS (.otf). It's a good idea to always decompose all glyphs (even the non-overlapping ones) in OTF, since composites in OTF are considered an "obsolete feature" and some rasterizers (e.g. Bitstream FontFusion) don't render them correctly. On the other hand, for OpenType TT / TrueType fonts, it's a good idea to keep the non-overlapping composites because the hinting is simpler and more consistent in the final font. With composite glyphs, FontLab Studio only needs to generate TT hinting code for the base glyphs and there are far less situations when the accented letter would render differently from the base letter. If you decompose glyphs in a TT font, some letters may render differently, and your font will be considerably larger. Overlapping contours, however, should be always decomposed and merged. If they're not, Mac OS X will render them so that the overlaps become visible. This is for example visible in Arial or Times New Roman on Mac OS X, for glyphs such as aogonek — you can actually see a thin white outline around the ogonek that protrudes into the "a". And yes, Ben Kiel's Python script for generating fonts is doing all the steps you're asking for at once. Regarding the encoding files — there is a section in the manual that discusses this, and you just need to follow it. An encoding file is a plain text file in which the first line has a special format that includes an encoding number (random) plus a name, while the subsequent lines are just glyphnames, one per each line, in the order that you desire. Once you put the .enc file in the right location, start FLS and switch your font to that encoding, you'll be able to use Glyph / Sort Glyphs / By Encoding to get the desired glyph order in your font. Cheers, Adam
