2007-06-12 01:26:26 • charles ellertson

Nick, you really don’t want to use the kern feature for positioning marks. Consider, if you kern a zero-width combining mark to the left to position it over a letter, the following letter will follow the mark and collide with the first letter. This is why the OTL mark feature exists: to position marks independently of kerning.

Regarding the contents of the ccmp feature:

This should definitely include any GSUB lookups resulting in precomposed diacritic glyphs that are not directly encoded in Unicode. So, for instance

n candrabindu -> ncandrabindu

would be included because there is no Unicode codepoint for the target diacritic combination. Unless the diacritic is going to be formed on-the-fly using mark positioning, then it must be mapped in the ccmp feature.

What about diacritic combinations that do have precomposed Unicode encodings? We’re in a transition period in support of these. Microsoft’s Uniscribe and Adobe’s CS3 apps perform buffered character-level substitutions for such diacritics, presuming that they are encoded in the font cmap. So, for instance, is a text string contains

a acutecomb

the shaping engine will check to see if the font cmap contains an appropriate entry for the /aacute/ glyph and, if so, use that.

So in theory one doesn’t need to include composition lookups for such diacritics in the ccmp feature. However, there is a risk that your font might encounter a shaping engine that does not automatically perform such character level substitutions, in which case it may be helpful to have them in the ccmp feature.

2007-06-13 00:11:29 • Nick Shinn

“ccmp” substitutes a base glyph and diacritic(s) by a precomposed diacritic glyph. [Or decomposes a precomposed diacritic glyph into base glyph and diacritic(s).]

“mark” provides positioning information for applications that can compose base glyphs and diacritic(s) ‘on the fly’. It defines attachment points (coordinates) first for the accent glyph, and then for the base glyph. This is like placing anchors in FLS, where, upon generating the composite glyph, the diacritic’s component is positioned such that its anchor point matches the base glyph’s anchor point. The “mark” feature however is different in that it does not lead to any actual precomposed diacritic glyph in the font – this part of the job is done by the application. “mark” only says how it must be done. (As to the FLS analogy, if one uses anchors anyway in FLS, then one already has everything needed to compile a “mark” feature.)

What would be really interesting from a font development point of view is using nothing else than ‘on the fly’ composition of diacritic glyphs, i.e. no ‘adieresis’ in the font any more but only one or two lines in the “mark” feature. But sadly, this is not possible since cmap require the existence of certain precomposed diacritics.

2007-06-13 02:01:56 • paul d hunt

Paul, maybe this will offer a glimpse. If your model is direct-entry typesetting, there is no absolute need for ccmp, because you can always put the character in via the glyph pallet. But let’s suppose you have an author preparing a manuscript in MS Word, or any other text editor that “honors” Unicode. S/he needs a “e” with a macron above and an ogonek below. Now the typesetter gets the author’s files and converts them to his/her typesetting program – in other words, we have moved from direct-entry typesetting to typesetting from supplied files.

The author could use a code for these characters, and the typesetter search & replace these codes with the correct character, But as long as you stay in Word before finally converting the files, a single replacement character would have to have a private use Unicode index, or you couldn’t later search for it (in Word, or any such text editor). If you search & replace with the correct Unicode string, the file can be searched without resorting to private use characters.

Or, the author could just type the Unicode string of “e” + “macron” + ogonek, which is unambiguous. Now, if the typesetter has made up the glyph(s) and defined the string in the ccmp feature, that string in the manuscript will call the correct glyph in the typeset file. Moreover, if the glyph is properly named (it has no Unicode number), it can be properly decomposed from a PDF file, in the unlikely event this is needed.

Finally, the character can be kerned, just like any other character. I don’t know of any limit to the number of characters that can be strung together with ccmp – though Adobe’s Type 1 (PDF format) will only work for a string of 12 numbers – i.e., decompose 3 Unicode characters. I don’t know if “mark” will allow for multiple diacritics. Even if it will, what if the different marks need different positions (anchors)? What about kerning, will it be blocked? Available for different kerning treatment following another letter – f + e versus f + e + (a marked) grave for example?

I’d look at all of this, and the various text processors before making a decision.

Paul: i was expecting him to say something about on-the-fly diacritical characters being defined by the mark feature.

That is what I said, although perhaps not clearly enough. Unless one is going to use the mark feature to build on-the-fly diacritics, you want diacritic combinations mapped in the ccmp feature. In other words, you have a choice about how to handle arbitrary diacritic sequences:

You can either build them on-the-fly using the mark feature

or

You can include precomposed glyphs and substitute them using the ccmp feature.

Paul: that’s not an answer to your question.

More accurately, just as with composition, there are two ways that decomposition can happen.

  1. Based on the Unicode values of the characters involved, not requiring OpenType layout. No relationship here to glyph names, except that Mac OS X still uses glyph names to determine Unicode values (at least for OpenType CFF fonts).

  2. Based on ccmp and one-to-many substitutions. In the font this is based on glyph IDs, but FontLab and AFDKO use glyph names for the source code to create the lookup(s).
