John, Thanks for posting the link to the white paper. I was asked a few questions offline on this, and I would like to share the answers with folks here.

Question 1: Are ‘name’ table IDs 21 and 22 now “official”?

  • The new ‘name’ table IDs were agreed upon with our partners, and their meaning is now well defined. Font manufacturers are encouraged to test their fonts with WPF font model, and I recommend adding the new ‘name’ table IDs for non-WWS families to resolve potential ambiguities and to make sure localized names are available for international clients. Please note that WWS families don’t need the new ‘name’ table IDs, they should set the ‘OS/2’ fsSelection bit 8 instead.

Question 2: Should Oblique font faces that set the new ‘OS/2’ fsSelection “Oblique” bit 9 also set the old “Italic” bit 0?

  • WPF font engine doesn’t depend on this decision, but I believe setting the old “Italic” bit is useful for compatibility with other platforms that don’t make distinction between Oblique and Italic font faces.

In addition, there are concerns about changing the layout of a document when a new “Italic” face is added to a font family that already has “Regular” and “Oblique” variants, or a new native “Oblique” face is added to a font family.

There are several things one can do to alleviate these concerns:

  1. WPF applications should use FontStyle = “Oblique” or FontStyle = “Italic” along with the appropriate FontFamily value (e.g. FontFamily=”My Font Family”) instead of specifying both family and face information in the FontFamily string (e.g. “My Font Family Italic” or “My Font Family Oblique”.) The former ensures that the application behavior will not be affected by Oblique faces that specify “Italic” in their subfamily string for compatibility with older platforms.

  2. Since adding, removing or modifying font faces on the system can generally change the layout of a document, applications or documents that would like to preserve the layout of a flow document are encouraged to embed relevant fonts, assuming the font licenses have appropriate permissions, of course. WPF has full support for font embedding for both fixed and flow documents.

Thanks for the feedback, and best regards, Mikhail

2007-04-25 20:23:34 • John Hudson

John, yes, your understanding is correct. Setting the WWS flag will bypass the font differentiation algorithm, and WPF will trust ‘name’ table strings and ‘OS/2’ values from the font file without trying to adjust them for consistency.

Before one ships a font with the WWS bit set, I recommend making sure names and style/weight/stretch values in the font are correct and consistent.

We tested the WPF font selection algorithm with thousands of existing fonts, and it is amazing how many fonts out there have inconsistent values. In particular, discrepancies between ‘OS/2’ usWeightClass and preferred ‘name’ IDs 16/17 (or 1/2 if 16/17 are not present) are very common. The goal of the WPF font differentiation algorithm is to correct these discrepancies for existing fonts, so that they can participate in the new font selection model.

I hope this clarifies why the new WWS bit is an “opt-in”.

Please let me know if you have further questions, and best regards, Mikhail
