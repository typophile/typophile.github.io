Todays font technology is not suited to handle such a massive amount of paths. It will slow down every font rendering engine and will turn into mysterious software crashes.

I would limit the amount of points to 1000 per glyph.

A. 1000 points = 250 dots = 15x16 dot matrix (15x16 pixel source bitmap graphic)

Maybe you can shift this value a bit.

B. 1200 points = 300 dots = 15x20 dot matrix (15x20 pixel source bitmap graphic)

To get more resolution in FontLab you should use an other font UPM size (usualy 1000em). Try 4000,6000 - maximum is round about 16000 or so.

An other trick is to use more glyph cells for one illustration. So you can enlarge the grid. On example A, two glyphs give you a dot grid of 30x16 (2x(15x16)).

When Andreas says “1000 points per glyph”, he means “1000 point coordinates”. Point coordinates in outline fonts must be integer, i.e. you can place a point at the coordinate (150,20) or (151,20), but not anywhere in-between. In other words, outline points are always placed on an invisible grid.

The UPM (units per em) size in Font Info controls the “fineness” of this grid. By default, you get 1000 grid units per glyph, but you can increase it to 2000, 3000, 4000 or even (max.) 16,000 grid units per glyph.

However, PostScript Type 1 fonts or OpenType PS fonts cannot have the UPM size larger than 4000 (effectively), while TrueType fonts can go up to 16,000.

Gradiate,

given the limitations described above, a suitable solution for you, of course, would be to use a rendering algorithm that creates circles that snap to those UPM units perfectly. Since 1 UPM unit in the font corresponds to 1 PostScript point in Illustrator/EPS, the halftoning algorithm that you’re using should only render the circles so that they’re 2, 4 6, 8, 10 etc. points wide (in diameter) and their centers should sit exactly on integer coordinates as well.
