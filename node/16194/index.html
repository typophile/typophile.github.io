2005-11-12 02:41:39 • Si_Daniels
I've just been playing around with some of the anti-aliasing settings in Flash 8. On the Properties Palette for text, there is a pop-up that lets you control anti-aliasing of text with the following choices:   Use device fonts   Bitmap text (no anti-alias)   Anti-alias for animation   Anti-alias for readability   Custom anti-alias... "Use device fonts" means that the system anti-aliasing (with settings controlled at the system level) will be used, but also means that the font needs to be installed on the system running the animation. "Bitmap text" is just what it sounds like. It works just as it did in the previous version of Flash. "Anti-alias for animation" looks like ordinary grayscale anti-aliasing. "Anti-alias for readability" uses sub-pixel antialiasing (like ClearType). I'm seeing some color fringing on my LCD Apple Cinema Display with this setting which I don't see in other programs. "Custom anti-aliasing" lets you tweak the anti-aliasing settings by controlling "Thickness" and "Sharpness". It also shows two values which can only be accessed via ActionScript--"Inside Cutoff" and "Outside Cutoff". I don't know what those mean (yet). Here is a simple sample I created: [img:sites/default/files/old-images/flash-aa1.gif] The first custom setting I did lowers the amount of color fringing, but looks fuzzier. The second custom setting shows what the maximum sharpness value does. Lots of color fringing. I don't know yet how the other two settings affect any of this.

2005-11-12 17:58:42 • edeverett
As far as I know, Flash does not embed fonts themselves. Instead it creates a set of symbols based on the outlines. In other words, it converts fonts to an internal graphic format. I don't think that should matter. Converting a font to another format, proprietary or otherwise, doesn't absolve you of your responsibilities around the font creator's wishes. This demo (http://www.kaourantin.net/2005/08/some-text-features-in-flash-player-8.html) seems to prove that they are storing enough outline and metric information for this to be considered a converted fully functional font. This is not a simple convert to outlines static image. Converting a font prior to embedding in a document is fine, and if that's what Macromedia/Adobe are doing and they're following the OpenType spec with respect to font embedding permissions then I don't see a problem. If however the claim is that this is not "font embedding" as defined in the OpenType spec then that changes everything. Then the font creators wishes around conversion and redistribution have to be respected, and as practically no-one allows this currently, someone wanting to use a font in Flash in this way would have to get permission from a font creator to convert and redistribute in this way. Converting a font prior to embedding in a document is fine, and if that’s what Macromedia/Adobe are doing and they’re following the OpenType spec with respect to font embedding then I don’t see a problem. As a font developer, I don't really have a problem with whatever they are doing because the font, however it or some representation of it is embedded in the .swf, cannot be extracted as a font, cannot be used outside the .swf, in short, is not being distributed "as a font." However, I can imagine a situation that would bug me. Say someone created an online application that sets type and generates a GIF or JPEG image for the user and delivers this as a file. And, let's say, this application includes a selection of fonts embedded within it, including one of mine. Such an application would seem to cross the line between application of a font (generally what users are expected to do with fonts) and redistribution of a font (a right font developers normally reserve for themselves and their agents). I would expect the developer of such a Flash application to acquire special rights from the owners of the fonts just as anyone who wants to bundle fonts with a desktop application. Online font preview test drives made in Flash (or other methods) come very close to crossing this line, but usually include some method (e.g., background patterns) to make it difficult or complicated to swipe.
