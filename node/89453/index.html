2012-01-27 11:01:28 • robster

Putting aside the licensing issue, OpenType features will continue to work if you don’t do anything that will change the GIDs of any of the glyphs referenced by the features – i.e. you don’t change the order of the glyphs within the font.

If you alter the glyph order, programs like FLS will recompile the features from decompiled source, and usually the features will work fine though occasionally you will run into glitches, particularly if the font includes features which FLS doesn’t recognise. This is usually only a major issue when working with non-latin fonts.

A small but potentially harmful side-effect is that you may end up with two different fonts (no matter how small the difference is) in your system, but with the very same font name and internal ID. I think it would be important to change the name of the font, jut to make sure the modified version is never mistaken for the original. Putting an .Md, or .Mod suffix after the style name in the PS Font Name, Full Name and OT Style Name fields will identify it as a different font, while still allowing it to appear as a member of the Family. You should also update the name in the TrueType Unique ID Record field as well.
2012-01-29 13:19:55 • .00

It’s certainly possible to make modifications to a font without changing anything else. Though, as Hrant said, the common font editors are not made to the purpose of “round-tripping” from font to work file back to font.

If you are familiar with ttx from the Python FontTools, you can convert both the original font and your modified font to an XML representation and find differences with one of the usual file comparison tools. But be warned, you’ll have to have pretty low-level knowledge of font formats to make any sense of it.
