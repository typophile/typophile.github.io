2014-12-17 16:56:51 • Core0

Neither of those sections are of much relevance to what you’re asking. The ‘unicode ranges’ are used to set flags in the font which indicate which unicode blocks contain characters – while some blocks are named for languages, it’s unicode blocks that are being referred to here, and while unicode has been expanded to include many new blocks, new flags haven’t been added for all of these.

The encodings section again doesn’t refer to languages, but to codepages which are largely obsolete since they are only really used by pre-unicode applications.

If you’re starting from an existing font there really isn’t any way to tell FontLan that you want to delete all characters used by (e.g.) Hausa other than to delete the unwanted characters, and then let FontLab automatically recalculate the two font info panels you mention above.

I’m not really sure what you mean by unicode ‘sets’. Unicode ranges are in many cases fairly arbitrary collections of characters and there’s no real need to ensure that any of them remain complete.

You mention you want to include Latin, Baltic (which is part of Latin) and Greek.

Here’s a brute force approach you might consider:

Open DejaVu in unicode ranges mode, then mark all of the glyphs in the following ranges: Basic Latin, Latin 1 supplement, Latin-Extended A.

In addition to the above, mark the following characters: 018F, 019F, 01FA-01FF, 0218-021B, 0237, 0259, 0275 and possibly 01C4-01CC and 01F1-01F3.

Next, switch to codepages mode and mark all of the characters in MacOS Roman and Windows Greek. (many of these will already be marked).

Next, look through the font and mark any additional characters that you want, such as mathematical characters etc.

Finally, select all unmarked characters and delete them. You should have a reasonable character set for Latin and monotonic Greek.

“What I would wish Fontlab could do would be that I click on a name on the unicode set, say, “Latin”, and it highlights the characters needed for it.”

It does not do that for Unicode ranges… but it does do that for encodings.

So it is pretty darn easy to select an encoding, color all the glyphs for that encoding, and then do that repeatedly across several encodings you want to cover.

NOTE that in the current app version, this is driven by glyph name rather than Unicode value, so when glyph names are other than what the encoding file expects, there will be misses. But otherwise, it’s a fairly quick way to do some things.

2015-01-11 15:56:51 • Michel Boyer

@Michel:

The digraphs in the list above are used for transliteration from slavic languages into latin and are compatability characters only needed to ensure round-trip transliteration. The characters with double grave and arch (“inverted breve”) aren’t core characters in any language, but are used (along with acute and grave) in croatian poetics (and possibly other languages). The characters with dot-below and dot-above are used primarily for transliteration of (primarily) semitic and indic languages.
