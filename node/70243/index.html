Although, there is sort of a workaround for FontLab. If you add a second axis, you can go from, say, light to regular along one axis and from bold to regular in the other. In order for this to work, you need to duplicate the “regular” master in both axes, which means you will “waste” a master and also have to make sure to repeat any changes you make in the duplicated master. It really gets to be impractical if you want to add, say, a width axis on top of this. Superpolator is much more flexible for working with interpolation.

2010-05-09 20:20:53 • blank

Workflow with Superpolator is based on .ufo, which is an xml-based exchange format for font development. You get the fonts in and out of FontLab using Python scripts (which are included with RoboFab tools and/or Prepolator).

If you have both FontLab and Superpolator running at the same time, when you update the .ufo from FontLab, it automatically is reflected in Superpolator. (This is also true of Prepolator.) You can also update, from within FontLab, glyphs that have been changed in the .ufo or import .ufo’s generated by Superpolator or whatever.

The only major problem is that reading and writing fonts to .ufo can be a bit slow, especially when you need to do a whole font with hundreds of glyphs, since it’s done with interpreted Python within FontLab.

My understanding is that the new version of FontLab (coming soon, I hope) will natively support .ufo, which should make working with external .ufo-based tools much smoother.

2010-05-11 00:52:06 • Michael Jarboe

MetricsMachine is very good. I find it makes me more thorough and systematic than when I do kerning in FontLab. If you do use it, it’s pretty much an all or nothing deal. You can’t very easily do some of your kerning in FontLab and some in MetricsMachine.

From MetricsMachine, you can export either to a feature file or an afm file. In the latter case, the kerning is expanded to simple pairs, rather than classes, and the number of pairs can be quite large. If you export a feature file (which makes more sense usually), you just copy and paste this into the OT editor in FontLab. There’s no kern table and you completely bypass FontLab’s class definition system, etc. There is also no need to worry about “key” glyphs (which are a FontLab thing, not an OT thing). Once you have the kern feature in FontLab, the only way you can see the kerning in FontLab is in the OpenType Features Preview panel.

It’s a very powerful app designed to do one thing really well–manual kerning. Some features I haven’t even tried yet (like “transformations”).
