2011-08-08 19:49:43 • JamesM
In handset type, as well as Linotype and Monotype, numerals intended for use in tabular matter (as in an annual report) were monospaced so that the columns would align regardless of which numeral you put in them. Monospaced numerals like this were referred to as “lining figures,” to distinguish them from “old style figures.” The latter have ascenders and descenders and variable set widths, and were designed for use within text paragraphs.

2011-08-09 02:10:27 • Claire Bibio

Early computer screens generally used monospaced character displays. It simplified the software and hardware, which was not very fast and had very little memory compared to modern computers. This “character mode” was usually the default. The character shapes were hard-coded into the hardware. Displays were advertised as having, say, an “80-column display”, meaning you could fit 80 characters across the screen (usually about 24 lines). The size was usually fixed, although sometimes there were “low res” modes that doubled the height and/or width of the characters.

In some computer systems (maybe most?) you could change out the default character designs to a set that was stored in memory, but this meant you had less memory for programs. Usually, you only did this for games or graphics programs.

Proportional fonts and other sizes were possible if you programmed the computer to use its “graphics mode”, which let you draw arbitrary images on the display and address each pixel independently (you could only address characters in character mode). However, this used a lot more memory and was a lot slower than character mode.

There were computers in the ’70s and early ’80s that had proportional fonts, such as the Xerox Alto and preview displays for computer typesetting front-ends. There were also graphics programs for the IBM PC and Apple II that let you work with proportional fonts for word processing and graphics. The Mac (1984) was the first popular personal computer to use proportional, arbitrarily sized fonts by default (no character mode–graphics mode only!), but it also included a monospaced font (Monaco). (I seem to recall that some reviewers knocked it for not having an “80-column display”.)

One thing that was probably also a factor in favor of monospaced fonts in early computers: Before CRT displays were economically feasible, you usually interacted with the computer using a teletype terminal. This was basically an electric typewriter that could transmit and receive character codes to and from the computer. It was natural (and technically easier) to emulate the monospaced typewriter output on the CRT display, rather than emulate typesetting.

2011-08-09 03:11:36 • Mark Simonson

Since typewriters produced tolerable text with monospaced typefaces, and since computer line printers (for technical reasons) were designed to produce monospaced text as well ([[http://www.quadibloc.com/comp/lineint.htm|theoretically]], if someone wanted, a chain printer with limited proportional spacing, like an IBM Executive, was a theoretical possibility, but you couldn’t do that with most other line printer designs like the more common drum printer), it would have made sense to design terminals with monospaced printing simply so that people could preview reports to be printed on printers.

As well, many computer languages had restrictions on what columns you could use for statements, because they were designed around the punched card.

But the clincher was in how terminals were designed. Because of the high cost of memory in the early days, a computer terminal would normally not have the ability to display arbitrary graphics. It would have storage to contain the characters to go on the screen, and use a “character generator” to look up which dots to display for each raster line of the character.

It would have involved a considerable amount of extra expense and logic to allow different characters to take up different amounts of space on the screen.

Of course, some terminals using unusual technologies, like the Tektronix 4010 or other storage-tube graphics terminals would have been an exception, but proportional spacing, unless the computer was used for typesetting, would have been considered a very wasteful luxury.

2011-08-10 05:09:20 • Mark Simonson

Monospaced typefaces certainly aren’t essential for programming. Algol and Pascal programs, for example, are often shown in a typeset “publication format”.

They had been useful for programming at one point simply because their output would be likely to be printed on high-speed printers that for technical reasons had to be monospaced.

And, as I noted, proportional spacing on displays would, in the past, have been an expensive frill.

That being said, monospaced typefaces still find some use in computing because many programs are used to produce output in tabular form, and a monospaced typeface ensures that “character position” and “paper position” are related, so that no special complexities are required in producing tabular output directly from a program that was written in a plain fashion, concentrating on its numerical calculations only.

Today, people tend to think of a programming language as a tool in a process of developing an application, which is given a fancy user interface for massive sale. At one time, programming languages were tools to get numerical answers to questions - used almost the way people use a spreadsheet.

2011-08-11 14:15:57 • Bert Vanderveen

Monospaced fonts are pretty healthy these days. If you start looking, you’ll find many variants and styles for different purposes. My favorite is DejaVu Mono (http://dejavu-fonts.org/wiki/Main_Page), which is not only well designed, but has very large character set, and is in active development.

Another reason, beyond those cited above, for the prevalence of monospaced fonts in programming, is that much of programming happens in a tabular mode. It’s common to have various levels of indentation for blocks of code, which are created using tab characters or runs of spaces. It’s also common to include comments that relate the code, but in another ‘column’ separate from the code itself. These conventions depend not only on monospaced visible characters, but perhaps even more on monospaced space characters.
