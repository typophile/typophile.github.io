well here is my script that i use. it does not need any glyph name list file to refer to. it transforms all the names to uni + the unicode number it has. it only works if you want the glyph names to be in unixxxx.

#FLM: Glyph name to uni pascal #pascal zoghbi

from robofab.world import CurrentFont

f = CurrentFont()

for g in f: uni = g.unicode if uni: unicode = “%X” %uni name = g.name lastBit = ""

	numberOfZero = 4-len(unicode)
	if name.count("."):
		k, lastBit = name.split(".")
		lastBit = "." + lastBit
	uni = "uni" + "0"*numberOfZero + unicode + lastBit
	g.name = uni

f.update()

2006-06-11 13:07:49 • dezcom

Chris, to address your background question: The Greek alphabet is essentially used to write a single language, so transliterated letter names can be used for glyphs without causing problems for developers who might call them something else. The Cyrillic script, on the other hand, is used to write more than fifty languages, and not all of them call the letters by the same names or even use them to signify the same phonemes. So codes are better for standard glyph names. That said, I use the uniXXXX form names for Cyrillic letters, not the old afii codes, since they are a bit easier to remember once one knows the Unicode set, are more efficient in terms of font size, and save me from keeping track of two different code naming conventions. You would only need to use afii names in Type 1 fonts with some pretty ancient software.

I use transliterated glyph names largely based on modern Russian names during production, and switch to the uniXXXX names for the post/CFF table entries. My Cyrillic production names are preceded by c. as a script indicator, which distinguishes e.g. Cyrillic c.A from Latin A.

2006-06-12 09:03:40 • titus n.

Chris,

I second John’s explanation. A simple example: in Russian, afii10068 (U+0433) is pronounced “g” (like in “good”) but in Ukrainian, this letter is used to write a dark “h” sound (something like in “hat”) while “g” is written using afii10098 (U+0491). There are other situations like that. A similar issue can be made between the Hebrew and the Arabic script. The Hebrew letters are practically used to write only one language so one could theoretically use “descriptive” transliterated names for the Hebrew glyphs, especially that the Hebrew letters have actual names (e.g. “aleph”). But the Arabic script is used to write many different languages of the world and the heavily diacritic structure of the script makes descriptive names at least illogical – you end up with names like “dotless beh”, which make little sense because you’re mixing visual and phonetical elements.

2014-06-18 14:11:18 • goloub

The only real reason is that in some corner cases with OpenType CFF fonts, encoding must be inferred from glyph names alone. Mostly this occurs when the font is printed to a PostScript file, and then that file is turned into a PDF. If the glyph names are not recognized by the PDF creator, the PDF will look correct, and print fine, but the underlying text will either not be present, or be incorrect.

afii names are not the only option, however. You can use uniXXXX names instead.

Or you can decide that this is an unlikely or irrelevant corner case and ignore the issue.

2014-06-24 13:20:24 • goloub

It’s not the PDF creator, but the workflow. Simply:

Use an OTF font. Print a PS file to disk instead of to a device. De-activate the fonts Make sure that your PDF creating software doesn’t know where the fonts are. Create your PDF. Open the PDF and see that everything looks right. However, copy-paste, search and so forth don’t work worth beans.

You can use Acrobat Distiller or anything else. It doesn’t matter at that point. I am not even convinced there is any need to test this—it is a pretty well-known and longstanding issue. It’s the main reason anybody cares about glyph names in the final, produced font. If you have a PS OpenType font (.otf), use completely non-standard names, and the original font is unavailable to the PDF creator, this will happen.

That said, obviously it is a corner case, for most purposes. So you may choose to ignore it, your call.

2014-06-27 09:42:01 • goloub

One thing I’m curious about is why Adobe has never actually remedied this since it would avoid a lot of headaches associated with CFF opentype fonts. Apart from the inconvenience of being forced to use the rather non-intuitive names defined by the AGL discussed in this thread, we are stuck with the need to duplicate certain glyphs (e.g. small capitals), to avoid various double-encodings which would simplify some fonts, etc. all to satisfy the limitations of a single program (Acrobat), and only in a fairly limited number of cases.

If Adobe were to update their PostScript drivers to embed cmap data into postscript files and distiller to make use of such data, these would all become non-issues.

The syntax adopted for including cmaps could use % delimiters, in which case older software would assume they were simply dealing with a lengthy comment and ignore the new data which would help ensure backwards compatibility with existing applications.

2014-06-27 15:38:16 • agisaak

People use old software sometimes. The issue would not go away for many years even if Adobe did update their PostScript drivers. Plus, one often hears that if people want to make the best and most reliable PDFs, they should be made with the source fonts available.

I started to write that still, it was not a bad idea, but then I thought about it more….

One problem is, it is not just the cmap. That is fine for one-to-one mappings of default glyphs, but not for alternates, ligatures, etc. There is all sorts of stuff that does not cover.

So for full reliability, you’d need to inject a LOT more info into the print stream. Possibly the rest of the font (the part beyond the CFF table), and that rest not subsetted down to just the glyphs used in printing. This would make for a much fatter and slower print stream, so it couldn’t be on by default, because the normal users who just want to print can’t get hit with a big performance penalty.

In which case, you would be in exactly the same boat as today, that sometimes things would go wrong, as long as people make PDFs from print streams. Even in a perfect future where that fatter print stream was an option.

I expect this is why it was never done. :(

2014-06-28 04:55:27 • dezcom

The issue about PDF creation isnt’t about Adobe. PDF is an ISO standard, and there are dozens of OSes, libraries and apps which create PDFs directly, using their own software: — Quartz, the OS X graphic subsystem — Ghostscript, a popular opensource package — Qt, a cross-platform application development kit — Cairo, an opensource graphic rendering subsystem — Apache PDFBox — libHaru — Apache FOP — GnuPDF — Global Graphics and many more.

PDF documents are written directly, or through an intermediate PostScript file, from desktop applications on various OSes but also from various libraries that are used on various web frameworks. PDF creation libraries are available in a large number of programming languages: C#/.NET, C++, C, Java, PHP, Python, Ruby, and others (in each of these languages, there is a large number of libraries available).

Hell, even some photocopiers allow you to scan a document, OCR it and save it as a PDF to a USB stick.

Since font embedding has always been a fairly complex issue, some of these libraries have solved it in as simple way as only possible.

All kinds of versions of these libraries are deployed on all kinds of devices, websites, server software, desktop apps etc. on different OSes worldwide. A fair number of these doesn’t even support OTF at all (just TTF), others support OTF but have gone a simple shortcut and treat them the same way as Type 1 (or more specifically, they just extract the CFF table from the OTF font file and put it directly into the PDF, without any parsing or subsetting, because it actually is the simplest way — but the only thing that at that time gets written into the PDF is the CFF table which relies on glyph names only.

So no matter what Adobe would do, it won’t “fix the problem”. Seriously — how many of you OS X users out there regularly use Adobe Acrobat to create PDF files, rather than using the PDF creation built-in into OS X? That PDF creation is not Adobe’s, that’s Apple’s. And it’s severely limited, too. :)
