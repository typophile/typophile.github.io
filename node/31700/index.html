If you’re going to have tabular figures, it stands to reason that other characters typically adjacent to numerals (e.g.: math operators, currency symbols, space, decimal points) should be a matching width so they won’t disrupt the grid and throw the tabulation out of whack.

Decimal points are a special case, since they are different characters depending on region and language, so some sort of contextual substitution would be in order for the comma, period, and space when adjacent to tabular figures as opposed to letters. Likewise for parentheses, brackets, and braces.

2007-02-23 06:52:53 • ebensorkin

If this is about tabular setting, some other considerations. I sometimes make up a comma.tab and a period.tab, not so much for their width but so they center better with the numbers – so the comma or period in 0,000 or 0.000 is a bit closer to the left digit than to the right. Never set financial statements, but I guess if I did, number-sized monetary symbols would be a plus.

In an OpenType font, I put in all the fixed spaces from Unicode 2002 through 200B. One of those is a “figure space” one a “punctuation space,” which I’ve always taken to be used for a period or comma in tabular setting – fits with the Unicode description. There are lots of other spaces there, and designing elements for tables to one of the sizes shouldn’t be too hard. This give you lots of ways to “pad” a table entry to keep decimal alignment.

When you write the features for tnum, just include the other glyphs you’ve made up for tabular setting.

OpenType features are for CS users. CS comes with a group of bundled Adobe OT fonts, which are the de facto standard. If my fonts behave differently than Adobe’s, people will think there’s something wrong with them.
