2009-11-08 12:15:07 • clankill3r

What exactly do you mean by ‘it doesn’t work’?

Do you mean you are getting an error from AFDKO, or do you mean that you aren’t observing ‘a’ being replaced by ‘bc’ when you test your font?

If the latter, this is not surprising. Very few major applications support one-to-many substitutions, so even if an appropriate lookup is present in your font, the substitutions will not be performed.

FontLab doesn’t support one-to-many substitutions, so you won’t be able to code such substitutions there. The Adobe CS 3 applications don’t support such lookups, so your font won’t work correctly in any of those (and AFAIK, CS 4 doesn’t support them either), nor do most other OpenType-savvy applications on the market today.

One-to-many substitutions are something which offers a great deal of promise for certain things, but they are still sufficiently undersupported that I wouldn’t recommend relying on them at this time.

Moreover, it’s really very unclear to me why you would want to perform the kinds of substitutions which you are proposing above within a font. Cases where I would consider one-to-many lookups useful would include things like

sub eacute by e acutecomb;

where the replacement of a particular glyph by a sequence of glyphs doesn’t alter the underlying semantics. The replacements which you are proposing, however, involve changes to the semantics which are the sort of thing which one would normally want to achieve by changing the character string rather than the glyph string, and that’s the sort of thing which should be done by the application rather than the font.

Perhaps you might clarify what it is that you are trying to achieve and why.

André

2009-11-09 15:57:14 • clankill3r

I don’t see how you could accomplish this within a font.

While one could in principle make use of one-to-many substitutions to replace English words with Dutch ones, it would be impossible to replace arbitrary English text with Dutch text. Because English and Dutch syntax are non-identical, you’d need to have code which handles entire phrases. Since the length of those phrases is unbounded you’d be looking at an infinite amount of substitution code.

André

2009-11-13 12:36:20 • Fontgrube

The OpenType font format does not support many-to-many substitutions. It supports many-to-one substitutions (they are implemented in FontLab Studio 5, and work in most OpenType applications), and it supports one-to-many substitutions (they are not implemented in FontLab Studio 5 but are in the new Adobe FDK for OpenType / AFDKO library version 2.5, which will be included in the next version of FontLab Studio and which is available separately from Adobe free of charge, but those substitutions don’t work properly in most Adobe applications).
