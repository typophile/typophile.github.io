Ben, you should consider where to position cased guillemets by looking at your face with French text, and then some other languages where the position of so-called “left” and “right” guillemets are switched!

2011-04-21 19:29:34 • Nick Shinn

In the flow of type you should use one position, even for caps etc. Ergo: centered on the x-height. In an all caps setting lifting the guillemets is acceptable.

My motivation: anything sticking out from the basic x-height should have a reason for doing so, and punctuation is NOT in that category.

There is even a case to be made for sticking the guillemets on the baseline. There is a reason why it is called baseline…

2011-04-22 00:23:53 • Té Rowan
The beauty of Opentype is that case specific sets of marks are easily possible. I always include height variations as classes so that the same opentype feature code picks up all supported variants. Since I always include case variations for hyphen, endash, emdash, paren, etc, adding guillemots is quite easy. It is in most cases a shift of vertical position only but there are a few times when you may kern slightly differently. I also make a variation for smallcaps and sometimes adjust size as well as position. I usually make the same sort of variation for currency, ampersands, and sometimes math symbols. The user, by selecting smallcaps, will automatically get the correct set to match so it is easy to use. The problem comes with some applications not properly supporting opentype features. This I can’t control. Smallcaps is tricky in one respect when you have mixed case so I just have the variants in place when Smallcaps from caps is chosen or in some cases I use contextual alternates so that it will only precede a smallcap glyph.
