2013-07-26 05:18:26 • jakemiller70
It is the wild west, in more ways than one. You don't need to encode glyph variants in your fonts, and I strongly recommend against using PUA codepoints for any text. In you case, though, it is a moot point, because all PUA codepoints have a presumed left-to-right directionality, so they are useless for Arabic. Adobe and some other developers have used PUA codepoints for glyph variants so that there was a way to access them in applications that do not support OpenType Layout. I think this is a terrible idea, because PUA codepoints are a kind of textual infection: once you introduce them to text, you lose the ability to search, sort, spellcheck, etc., and if you try to display the text in a different font there is no way to know what glyphs you will get, if any, for the PUA codepoints.

2005-12-18 19:13:39 • Nick Shinn
In an Open Type Arabic font, you don't need to encode the variants at all. The variants will be accessed through OpenType Layout features, which use glyph IDs in their lookups. So the underlying codepoint in text for the nun will always be U+0646, regardless of the variant shape used and the layout features applied. Most of the Arabic presentation form codepoints in Unicode were a political inclusion that provided a temporary method of doing basic Arabic shaping while waiting for smart font technology like OpenType, AAT and Graphite to come along. If applied in a pure way, the Unicode character/glyph model would have prohibited inclusion of any presentation form codepoints: all you should need are the characters for the Arabic letters. Everything else can be handled at the glyph processing level. I tend to agree with you about Arabic 'ligatures'. Have you read Tom Milo's essay in the book Language Culture Type? It is an excellent analysis of the Arabic writing system and mounts a compelling argument against the whole notion of ligatures in Arabic. You might find my essay in the same book helpful in terms of understanding OpenType language processing (although I'm still embarassed about the Arabic-related typo in one of my examples: I really do know the difference between kaf and lam!)

Actually, there is arguably one true “ligature” in Arabic: the lam-alif combination. It is interesting that some of the earliest Islamic sources refer to the lam-alif combo (pronounced “laa”) as a single character of the Arabic alphabet! When talking about type, especially in regard to smart digital fonts, I tend to use the term ligature to refer to a specific display solution, i.e. the rendering of more than one character as a single glyph. This is the level at which analysis of the Arabic script and implementation meet, and hence where it gets interesting. If one's analysis of the script is that the form of each letter varies according to adjacent context, i.e. is determined by the particular shape of the preceding and/or following letter, then one can describe the lam_alif 'ligature' in the same terms: this is the form that the lam takes when followed by alif, and this is the form that alif takes when preceded by lam. What confuses the analysis is trying to determine where the lam ends and the alif begins, which is probably why this combination has been taken as a singular textual element. But the analysis is valid enough for some fonts to implement the writing of lam+alif using two glyphs, rather than one ligature.
2005-12-19 20:36:51 • John Hudson
What confuses the analysis is trying to determine where the lam ends and the alif begins, which is probably why this combination has been taken as a singular textual element. Actually, the primordial reason is more cosmological/numerological. The word [laa] (symbolized by [laam]-[alif]) means "no", as in, "No god but Allah". The symbol for the negation is counted as a single "letter" of the alphabet. It is also significant that the Arabic for "No god but Allah" is composed only of alif's and laam's (with a haa at the end). Phenomenologically "laa" symbolizes the stage of mystical or unitary consciousness where all discrimination ceases. At the same time the name for this character/letter is just [laam-alif]. The point is that no other letter combination is reified as such in Classical Arabic other than this one: It is often and traditionally counted as the second-to-last letter of the alphabet. Thus one could make a strong argument for maintaining it as a ligature in the Unicode main Arabic block. [laam-alif] is in the interesting (and perhaps unique) situation of being both a ligature and a single character at the same time Of course, for type design and typography purposes this is all practically irrelevant-) But on philosophical grounds I would place U-FEFB in the U-0600 block... It depends on what you intend your font to be used for. For example, if your font is going to include a large number of mediaeval variants forms, ligatures, and symbols, it might be a good idea to follow the standards laid out by a group so that your font can be used on all documents that previously used theirs, and vice versa. Basically, PUA is a sort of neutral-zone, where anyone can define their standard using standard code point references (as opposed to adding encoding slots which most likely won't be reproducible in all applications). A number of groups have collaborated to define these spaces but they are hardly required. For an example of how these work try reading more about the Medieval Unicode Font Initiative at http://gandalf.aksis.uib.no/mufi/ . On the other hand, if you're including dingbats, ornaments, or other features that can't or have no need to be transcoded (transliterated? transgraphemed? transcoded?) to another font, then feel free to fill it up as haphazardly as you'd like. «El futuro es una línea tan fina que apenas nos damos cuenta de pintarla nosotros mismos». (La Luz Oscura, por Javier Guerrero)
