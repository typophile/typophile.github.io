2008-10-22 18:57:49 • Goran Soderstrom
Goran, I'm quite surprised that you're saying it ever worked. I don't remember that it ever worked. The problem is: imagine an MM glyph with two squares (so there are 8 nodes in the glyph). In the first master the squares are small and do not overlap, in the second master the squares are larger and overlap. A perfectly normal case. Now if you'd try to remove the overlaps, the first master should not change but in the second master it would have to remove some nodes, so you'd end up having incompatible masters. This is why remove overlap in FLS is not MM-compatible. I agree it would be nice if the function was written so that it removes overlap in each master and then blends the masters together, and "fails" only if the final masters are not compatible. I'll suggest that to our developers. In the meantime, I suggest that you leave the overlaps in your MM and remove them only in the generated instances. Adam
