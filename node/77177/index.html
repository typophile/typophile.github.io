The clipping in Word is related to a performance optimization that treats each line with in a paragraph as independent when it is rendered to the screen. The “empty” space between lines is drawn as white, which can be drawn over other lines – and therefore clip those lines. When you print, the empty space isn’t drawn so hence no clipping.

Word linespacing used to be fairly straightforward: à la other MS text apps like Wordpad and NotePad, the default linespacing was determined by the combined OS/2 WinAscent and WinDescent metrics, as rounded to logical units for individual ppem sizes.

In Word 2007, MS made two significant changes:

  1. They set the default linespacing to be 1.15 times the vertical metric in the OS/2 table to increase the default leading. Typographically, this is an improvement in most Word documents with most fonts, since most such documents tend to have quite long line lengths relative to type size. [They also increased the default text size from 10pt to 11pt.]

  2. They implemented support for recognition of the OS/2 table version 4 fsSelection bit 7, which tells an application to use the OS/2 Typo metrics for linespacing instead of the Win metrics. In theory, this means that a font can have independent linespacing and clipping metrics, which means that fonts with some very tall variant glyphs, e.g. Gabriola, can have adequate non-clipping areas defined by the Win values, while not needing excessively large default linespacing. But this is where things get confusing and difficult to predict, because there seems to be a Word linespacing algorithm that in some situations produces linespacing that is independent of the font metrics. It looks like the Office developers implemented support for fsSelection bit 7, saw some fonts in which they didn’t like the result of using the Typo metrics, and decided to bypass the metrics and apply algorithmic linespacing. This means that in some fonts the Word linespacing is defined by neither the Win metrics nor the Typo metrics, but by Word, in a way that does not appear to be publicly documented.

As noted in my response above, whether clipping occurs depends on how the user specifies linespacing in the paragraph properties dialogue. If linespacing is specified as a multiple of the basic spacing, then clipping will occur if the specified value results in spacing that is tighter than the Win metrics (or, I suspect, tighter than Word’s algorithmic spacing). But if the linespacing is specified as an absolute value, then clipping usually won’t occur, even if the resulting linespacing is equally tight.

I have not looked at Word 2010 linespacing to see if it is the same as 2007.
