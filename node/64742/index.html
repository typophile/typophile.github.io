There is no dilemma.

When choosing “binary”, FLS will simply include OTL tables “as is” in the OTF to be generated. In this case, the glyph set and glyph order better not be changed after having imported the font, for the simple reason that OTL tables are glyph index based.

One may blame FLS for strange behavior, but not in this case.

@ Karsten

I have experimented a bit with TTX 2.2 (contained in the AFDKO 2.5 21898) and FLS 5.0.4.

  1. Reset the options of FLS to default.

  2. Open MinionPro and generate MinionPro-FLS-compiled. Choose the compile option, when you are asked.

  3. Change to the index mode and substitute a and b. Generate MinionPro-abSubstituted-FLS-compiled.

  4. Decompile both files with TTX.

  5. Compare the files. You will find changes in three tables (except from the table, in which the checksum is stored): GlyphOrder, GPOS and GSUB.

  6. Now I have played a bit around. If you decompile the original Minion and substitute a and b in the GlyphOrder table only, you will either get an invalid font or the order will be reset, dependent from the way, in which you recompile the font. (Decompile the GlyphOrder table only, substitute a and b and merge the GlyphOrder table into the original font file; or decompile all tables, substitute a and b in the GlyphOrder table and recompile the font.) But forget this for a moment.

If you substitute a and b in the GlyphOrder and the GPOS table, the recompiled font works. And although the positions of a and b have changed – check it in FontLab or decompile the GlyphOrder and the GPOS table of the compiled font again –, the small capital feature seems to work correctly in Indesign. “a” is substituted by “a.sc” and “b” by “b.sc”.

Do you have any explanations for this behavior?

It seems to indicate, that the GSUB table is independent from the GlyphOrder and the GPOS table. May it be, that the GSUB table uses its own (internal) glyph order/index?
