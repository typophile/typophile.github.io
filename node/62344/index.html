This is informational. If a major software developer has used PUA codepoints for some particular purpose -- most likely legacy -- then there is a risk that these codepoints might come with assumptions attached in that software, and it would be a good idea to avoid using these particular PUA codepoints. Alternatively, if you specifically want to make a font that is compatible with the use of PUA in particular software, then you want to know what this use is. And it would be perfectly possible for FL to expose multiple and conflicting uses of PUA codepoints in this way. All you are looking at is a labelling system.

2009-09-24 20:38:10 • Theunis de Jong
See page 11 of this Unicode PDF: http://unicode.org/book/ch13.pdf Unless one actually has a need to produce a font that is compatible with a particular company's use of PUA codepoints, it is best to avoid that relatively small set of codepoints, just in case some piece of software is making assumptions about those codepoints. That said, the nature of the PUA is explicitly such that neither software developers nor font developers should expect any degree of safety in making assumptions about the use of any PUA codepoint beyond their own internal use. The basic rules of the PUA apply as much to the ‘Corporate Use Sub-Area’ codepoints as to any other PUA codepoints, so there is nothing to prevent another company or an individual using the same codepoints for other purposes or even claiming them as their own ‘corporate use area’. Corporate use of the PUA has occasionally been an issue for me in my work, but only because I'm making fonts for companies that have already made use of some PUA codepoints and I need to make sure that I use fresh ones that their software does not associate with other characters.

I don't think there's any nefarious plot going on here - but then again, I am from SIL. I'm actually surprised to see this documentation of the SIL PUA ranges in FontLab. We didn't ask for it. It looks like Fontlab is only trying to serve their customers by gathering together info on the most common uses of the PUA. SIL's use of the PUA is [[http://scripts.sil.org/UnicodePUA|publicly documented]] and follows the guidance of the Unicode Consortium. Like John, we try by all means to avoid using the PUA. Most often it has been used as a practical temporary way to encode language data during the sometimes lengthy process of getting new characters accepted into Unicode. Around 80% of our PUA assignments are characters that have now been accepted into Unicode, and the use of them as PUA characters has now been deprecated (no longer recommended). Our detailed [[http://scripts.sil.org/PUA_deprecation|deprecation strategy]] is available, and explains why we keep characters at both new and old codepoints in our fonts. The other 20% are characters that Unicode rejected, ones that we have not yet proposed, or ones we ourselves have later deprecated. FontLab's acknowledgment of the most common PUA uses is not advertising. They are only trying to let font developers know about them so they can avoid those places if they have customers that already use MS, Adobe or SIL fonts - or use them in the same way. Note that since this is a Private Use Area you can feel free to use the range however you choose. Nothing bad will happen if you assign random chars to fall in the 'SIL PUA'. :-)

Andreas, a massive number of the PUA-encoded glyphs in your excellent survey are presentation forms of letters plus diacritic marks that have standard Unicode encodings using combining mark sequences. While I understand that there have been difficulties in the past in reliably rendering combining mark sequences, these difficulties have very largely disappeared, and maintaining non-standard encodings for these diacritic letters is very bad text processing and interchange practice. All you need is one user receiving the text and either displaying it in a font with different PUA usage or being unable to display it at all, and the whole ‘practice and agreement’ falls apart. Standards exist for a reason, and as soon as something is encodable via the standard private encodings should be deprecated as quickly as possible.

If you place your own copy of "uranges.dat" in the Data subfolder of the FontLab Studio shared user folder, FontLab Studio will automatically use that. Feel free to customize that file and remove any range labels that you feel "offensive". This is precisely why FontLab Studio is very customizable. Since Adobe, Microsoft and SIL have publicly documented their use of the PUA, we are providing their assignments so that users of FontLab Studio who would like to adhere to their "quasi standards" can easily access them. You can publish your own PUA mappings in the "urange.dat" format on your website so that users who would like to take advantage of your use of the encoding space can easily do that. I do not consider our information any more "promotional" than the fact that we use the Adobe Glyph List for New Fonts rather than providing completely invented glyph names -- because there is software in the world that actually makes use of those mappings. We try to be compatible with major software vendors' products so that the fonts our users generate can be, too. That's it. Regards, Adam
