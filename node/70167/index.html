2010-05-08 04:32:04 • Ray Larabie

If you’re talking about the Mac Roman char set and no kerning classes, then you’re essentially asking what is the average number of kerning pairs in a legacy Postscript Type 1 font.

Still, it depends. It depends upon the style of the design, the effort put into shaping and spacing to obviate kerns (not necessarily a good or bad thing, just a design intent), whether the design is intended for display or text, and the extent to which a designer chose to go (kerning accented chars or not, which punctuation combinations to give attention to, etc.)

In the original Whitman text fonts, there were around 1800 individual kern pairs. In the Display, more like 5000.

I once did an analysis of some rough breakdowns. I can’t find the Whitman analysis; but there is this old thread, with data from a few other fonts:

2010-05-08 13:52:26 • kentlew

By the way, the reason there are so many kerns in Williams Caslon is because its glyph set is very large, including CE characters, Bold Italic small caps, swash characters, and 6 styles of figures. You have to do “exceptions” for the accents, and the swash are kerned to both lower case and small caps. So everything explodes exponentially with more characters that need to be kerned with one another.

Also the italic follows the varied slants of the original (though all is made 3% more upright). I think because of the varied slants, it took a lot more kerning to get the evenness of a modern italic. There are about double the kerns in the italic as the roman.

Also this follows the FB method of not putting negative side bearings in vwxy, so that they won’t touch even if kerning isn’t operating. This creates a need for more kerning to get the diagonal characters even in the final.

Some apps don’t support class based kerning. So even basic stuff like commas, emdashes, ellipsis, base quotes and smart quotes would end up unkerned in those apps if the class kerning wasn’t at least somewhat expanded. For all caps fonts, unexpanded kerning meant the “lowercase” set ended up with no kerning at all. For the first few years of OTF, MyFonts samples didn’t display class based kerning. If you didn’t expand the kerning, I’d only see the class parents kerned.

Now, with web embedding, class kerning isn’t fully supported so, if I don’t expand kerning, it won’t kern.

In earlier OTF times, there was an OS9 app that supported OTF but not kerning classes. I can’t remember which one but a few customers mentioned it so I made sure I expanded everything.

In Windows there was something . . . Corel Draw? I think there was a version that supported OTF but not classes.

2010-05-09 09:45:33 • Ray Larabie

what’s the technical limit for the number of kerning pairs in an OTF?

There’s no simple number. The format is limited by the size of a subtable (64kb), though we now have the “extension” mechanism to work around that, in newer environments. Some individual applications will start to misbehave if some limit is exceeded, but the nature of that misbehavior and the size of the limit can vary by app and even by app version. Some applications have no limit whatsoever (for example, InDesign).

Examples of different kinds of misbehavior:

  • only recognizing the first n pairs
  • only recognizing the last n pairs
  • the app crashes
  • the app fails to recognize any kerning in the font

Generally, Adobe notified all the application developers whose apps they saw misbehaving, about 8-10 years ago. I know that Framemaker and QuarkXPress improved.
