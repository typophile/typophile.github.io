I've had this message -- but not the phenomena you report -- happen to me. In every case, I have my classes defined in both the *class* window and the *features*. If there is a disagreement, you get the message. Pick one or the other. Since I write the features file off and use a text editor on them to do further work, I pick *features* as primary, and have all my classes there. When I get the message you got, it means I haven't cleared the separate *classes.* (with FL5 on a PC, I believe it is: window > panels > classes, then "reset".) Alternatively, if you want to keep the classes file (panel), I suppose you should not also have them in your features file. Try this on a copy first, of course.

Not even if there is a disagreement. When compiling feature definitions, FontLab Studio automatically copies the class definitions from the Classes panel to the OpenType panel. But if there are already class definitions in the OpenType FEA syntax in the lower-right part of the OpenType panel, then each class is being defined twice. If you don't need the class definitions in the OpenType panel, just remove them from there. Formally speaking, the title of this thread is misleading. The thread is called "Strange errors" and the poster writes "I get this error message", but then cites a message that clearly says "[WARNING]", not "[ERROR]". So it's not an error message, it's a warning message. A "warning" is not an "error". In FontLab Studio, an error message in the Output panel is prefixed with "[ERROR]", while a warning message is prefixed with "[WARNING]". In most applications, errors lead to a technical fail while warnings lead to a technically successful completion. Warnings are issued when the application can complete the process but its internal logic indicates there "may" be something wrong, so it raises a potential problem to the user, and it's up to the user to decide whether it's a "real" problem or one that the user can ignore. Errors are issues when the application cannot successfully complete a process at all. Based on this, it is evident that when it comes to seriousness, a warning message is less serious, and an error message is more serious. A.

Regarding XeTeX, there are some bugs in its underlying ICU Layout library, but I don't think any that would produce this kind of behavior. Did you use fontspec to access the font? If so, try accessing it directly (using plain XeTeX, not XeLaTeX) as in: \font\samplefont = "[/Library/Fonts/Cardo.otf]" at 72pt \samplefont and see if the problem also occurs. If it doesn't, it would indicate some fontspec caching problem. Adam
