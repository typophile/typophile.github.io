Embedding a "true" bitmap font into a TT shell is not enough - MacOS doesn't like those either. Especially not grayscale ones, which can cause even pre-X systems to perform a major flaming crash! Ironically, everything used to work seamlessly before System-8. The only way -like Chris says- is to make an outline font that behaves like a bitmap font at one specific size. These are generally called pixelfonts (with some nice automation indeed provided by Pyrus), and the great news is they can be grayscale too (blowing away any other font rendering method). hhp

> distinction between bitmap fonts and pixelfonts It's not like the terminology is solid, but there are three kinds of screen-optimized fonts: 1) The superhinted stuff, like the MS core fonts. These have no bitmap data - they have special code ("instructions") to perform funky distortions to the outlines which makes them render out as nice bitmaps at each size. These are always 1-bit (b&w) in practice, although: it's actually possible to superhint for grayscale rendering - it just seems to be too much work; and above certain PPEM sizes grayscaling is enabled, essentially because the rendering algorithm no longer needs much hand-holding. 2) What are generally called pixelfonts, which are regular outline fonts that have blocky forms that map to a specific pattern at a given size. These can be 1-bit (b&w) or more recently grayscale. 3) "True" bitmap fonts, which contain actual bitmap data, either as stand-alone files (like FON, or the old Mac NFNT stuff), or embedded in TT fonts which may or may not have outlines too. These can also be grayscale in theory, but it's hard to make them work on Windows, and since System8 MacOS has flaming-crashed if you try to use one. Sad. Although #3 in its embedded flavor is the most elegant (not least because it provides a nice separation between print and screen rendering), it's looking like you want to avoid it, especially on the Mac (although doing 1-bit versions for Windows is great*). And even #1 is being demoted on the Mac since Panther seems to ignore hinting. So #2 is looking great overall - even if they try the OS makers will have trouble preventing it from working. * http://www.themicrofoundry.com/f_arasan.html Then there's ClearType, which is a different kettle of fish. Technically it's possible to apply any of the above three classes to CT (except maybe not the grayscale flavor of #2) but I've yet to see anybody actually do it. hhp
