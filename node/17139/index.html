Opentype: replacement and positioning of sups, sinf, numr, dnom doing the Adobe way?

January 8, 2006 • Sebastian Nagel

I am trying to get the opentype features sups, sinf, numr and dnom (superscript, etc.) to work.

The glyphs 0-9 look exactly the same, so my idea was to use a sub- and a pos-command:

  • first substitute the original big numerals with smaller variants
  • then reposition them vertically, depending on whether they are superscripts, inferiors, numerators or demoninators

But Adobes Garamond Pro is using 4 instances of the small variants, each of them identical, but with a different position, and doing everything via substitution. Is this the “official” way to implement the features? What is the advantage?

An entire thread about this is at http://typophile.com/node/16879. ;-)

There is no “correct” way. It depends. If you would like to kern “sups” or “numr” or “sinf” numerals with preceding glyphs, it would be better to use 4 instances of the smaller numerals together with sustitution commands. (Kerning values for superiors are not necessarily good for inferiors too.) If you don’t intend to kern them, you may use sub+pos. This helps keep the character set a bit smaller. However, pos and sub in the same feature can be implemented only with VOLT so far, not with ADFDK and thus not FontLab Studio.

Provided that this is the way MS and Adobe want to go in future, then it may be wise also to include a “subs” feature with the same sub/pos commands as “sinf”.

The benefit of using separate glyphs for each feature, rather than one set of glyphs and a GPOS vertical position adjustment, is that in the latter case you cannot reliably kern to adjacent glyphs. Consider, if you have an uppercase T followed by a subscript 3, for whatever reason. You probably want to slightly kern that subscript number closer to the T, but if you then use the same glyph in a superscript position the same kern data will cause it to collide with the top bar of the T. Since kerning is simply a horiztonal relationship between two glyph IDs without any knowledge of their relative vertical alignment, there is no way around this via the kern feature.

Now, it may be possible to include pair adjustment GPOS lookups within the specific script style feature. So, for instance, you might have a lookup that kerns the script style 3 to the T in the subs feature. In theory, this would work. But it is simpler, and also faster in terms of processing, to add the extra glyphs and do it the way Adobe and others have. Glyph space in fonts is pretty cheap, especially if you use composites or subroutinisation, whereas every additional lookup in a font slows down processing and layout, and GPOS lookups are slower than GSUB.
