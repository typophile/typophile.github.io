2006-03-02 12:43:48 • vinceconnare

Kern classes have their advantages. The classes result in a compressed size of the kerning information in the font and they also make it easier to keep an overview over kerning pairs during production. The latter can at the other hand also result in an unwanted levelling/simplification of the kerning information, because the resulting groups can contain pairs that actually need a (slightly) different treatment. Some grouping can also demand a very well organized standardization of the spacing of fonts.

I had a brief look at Chester’s features file for Apex and I noticed that for instance P and Thorn are placed in one group. I don’t know if it works in this case, but the chances that the Thorn needs the same kerning as the P are small, I think. I was also wondering what happens with the accented variations of the a when the T_a or V_a combinations are kerned –90. With this negative value I would expect enumerating pairs for the a’s with accents, but I could be completely wrong, of course.

I have actually no idea how kerning and especially the kern class files are handled by FontLab, but from what Chester writes I conclude that the kern class file must match the font’s glyph set in some way. In DTL FontMaster the stuff is actually handled in a quite straight forward manner: in KernMaster one can select a kern class file and a file containing the kerning pairs and the output is a kern features file. Because everything in FM is batch oriented, the classes and pairs can be used for generating kern features files for a series of fonts at once.

For professional font production a reproducible result is one of the most important things. In FM for instance the OpenType features output is controlled by standardized features files (Adobe syntax). If the features are updated or enhanced, the improvements can be applied to all applicable OT fonts by generating new versions in batch in DataMaster. Also reproducible kerning is of importance. I don’t think that there is better software for this purpose available than KernMaster. After generating the kerning pairs, sometimes it might be necessary to make some minor improvements by hand, but overall KM will handle the most complex situations, including the kerning of Cyrillic and Greek extremely well.

I shall add that since version 4.0, FontLab has been using the Adobe FDK for OpenType code to generate the OpenType Layout tables, including the “kern” feature. So from the same .fea syntax defining OpenType Layout features you will get the same binary tables – no matter whether you use FontLab/FontLab Studio, DTL FontMaster or the stand-alone Adobe FDK for OpenType.
