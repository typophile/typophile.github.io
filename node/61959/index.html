Let’s say I’ve defined the capital A as a kerning class for all accented A’s. Like this:

_kern1: A' Adieresis Aring Agrave Atilde Acircumflex Aacute Amacron Abreve Aogonek

How can I apply the left A kerning to the AE? I’ve always just done a copy from A and paste special, left kerning on the AE. It works, but I’d like to do it with classes

I can see that I can define a whole new kerning class and have it apply to only the left or right side. But the A already has a class defined, I can’t define another class for A: it already has a class.

If I understand you correctly, this is a simple task:

_kern1: A' Adieresis Aring Agrave Atilde Acircumflex Aacute Amacron Abreve Aogonek with LEFT checkmark marked: [√] | [ ]

_kern2: A' Adieresis Aring Agrave Atilde Acircumflex Aacute Amacron Abreve Aogonek AE with RIGHT checkmark marked: [ ] | [√]

The (confusing) logic behind this is to understand if the class is the first or the second letter in the kerning pair. For this reason I always change the name of the kerning class:

_Afirst: A' Adieresis Aring Agrave Atilde Acircumflex Aacute Amacron Abreve Aogonek

_Asecond: A' Adieresis Aring Agrave Atilde Acircumflex Aacute Amacron Abreve Aogonek AE

———

And you might want to include Æ in a E class as well:

_Efirst: E' Egrave Eacute Ecircumflex Edieresis AE (and so on) with LEFT checkmark marked: [√] | [ ]

Is this what you are looking for?

2009-09-14 15:01:19 • Cristobal Henestrosa

There are two checkmarks that represent the glyph classes in a kerning pair, and the dot in the middle represents the kerning pair.

If the left checkmark is enabled, all glyphs in the class that is on the left side of the kerning pair get the same kerning value as its key glyph.

If the right checkmark is enabled, all glyphs in the class that is on the right side of the kerning pair get the same kerning value as its key glyph.

The “AE” glyph is one where you want to share E’s kerning value if the glyph appears on the left side of the kerning pair — so you enable the left checkbox.

Cheers, Adam

2009-10-07 01:01:52 • Cristobal Henestrosa

Cristobal,

in left-to-right scripts, if glyphs are identical on the LEFT side then it means they will share the same kerning value if they are the 2nd glyphs in the pair. In right-to-left scripts, if glyphs are identical on the RIGHT side then it means they will share the same kerning value if they are the 2nd glyphs in the pair.

In both cases, the 2nd checkbox needs to be activated. It would be far too complex from the UI standpoint to somehow make sure that the checkboxes relate to the left or right side of the glyphs, because that depends on the direction of the script. But if you think about the logical order of the glyphs in the kerning pair, one is always 1st and the other is always 2nd, and this is what the checkboxes reflect. I’d be happy to hear your suggestions how to improve this, provided that the solution will be workable for both LTR and RTL scripts, of course.

Cheers, Adam

2009-10-12 02:02:28 • Ray Larabie

Classes don’t need to be numbered. That’s just what happens by default.

You can use any naming scheme that makes sense to you, as long as you start it with an underscore. The underscore tells FontLab that it’s a kerning class, and not a metrics class (which start with a period) or an OpenType class (which have no special mark).

Incidentally, if you decide to bypass FontLab’s class kerning support and create your kern feature some other way (such as writing kern feature code directly in the OT panel or using an external tool such as Metrics Machine) you don’t need the underscore or the key glyph (the one with the tick mark) at all. Those are only for FontLab’s purposes and are not part of the OT spec.

2009-10-12 15:49:31 • Mark Simonson

Mark,

class names are not at all part of the OpenType spec altogether. They’re not included in the final font files at all. They’re just being used by the Adobe FDK for OpenType library that converts the text-based feature specification into binary OpenType Layout feature tables. And yes, indeed, the convention of starting names of classes that are used for kerning with an underscore ("_") is just a convention used within FontLab Studio. This way, FontLab Studio “known” which classes are to be used when it generates the feature specification for the “kern” feature.

Similarly, the key glyph is a concept used within FontLab Studio to simplify the internal storage of data (kerning pairs are “attached” to glyphs in FontLab Studio). This is pretty much a remainder of old FontLab 3.x where there was no concept of class kerning. It’s possible that at some point, we’ll develop a fully new way of dealing with kerning which will not require key glyphs.

Cheers, Adam
