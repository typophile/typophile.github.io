Glyph naming standard – a proposal

September 29, 2007 • filip blazek

I am not a type designer but I work with fonts almost every day. I help several type designers to build accented characters, class kerning and OT features. During the last four years I've came up with dozens of font families from different designers. Every type foundry has its own glyph naming scheme and different suffices. There are many type designers from many type foundries here on Typophile. I think we should together make a well prepared glyph naming scheme and then force all the important players in the font industry to accept it as a standard. I am sure we all would benefit from such initiative. If the glyph names would have standardized suffices, FontLab (or third party developers) could offer tools to build OT features based on glyph suffices and glyph names. It would be easier to work with encoding files etc. It could also help programmers, their scripts and testing routines would be accessible to a broader group of users without the need to modify the code; also the scripts designed to build OT features based on glyph suffices would be easier to write and use. The font production would benefit as well: it would be easier to copy features from one font to another etc. etc. This new standard could be used in two different ways. I can imagine FontLab would allow users to switch between production glyph names and standardized glyph names. During font export, the glyphs could be automatically converted to proper glyph names (uniXXXX, afiiXXXX). What should we do? 1. Standardize glyph suffices. We could either follow Adam Twardoch recommendation to use suffices based on standardized code for OT features, either we could use more user-friendly suffixes (sups v. superior). We should discuss this. 2. Standardize glyph names with two or more suffices. For example alternative form of small cap A (a.small_alt, a.alt_smcp...). 3. Standardize glyph names for ligatures. Should we use ffi, f_f_i or both? Should we assign Unicode indices to ligatures? How to name ligatures composed of alternative forms of glyphs? This should be discussed. 4. Create user-friendly names for Cyrillic and other glyphs. I understand that to name Cyrillic glyphs could be a difficult task as not all the users pronounce the letters the same way, but I am sure type designers would benefit from this standard (even if the standard would only be for production). Also many other glyphs should have names: arrows, letter-like symbols, math symbols, dingbats... 5. Create a "Typophile Glyph List for New Fonts" and offer type designers tools to covert old glyph names to this new standard and tools to convert this proprietary encoding back the current standards. This should be accompanied with a document explaining problematic glyphs, their names and Unicode indices (scommaaccent v. scedilla, fi v. f_i etc.), language specific problems (ligatures and small caps in Turkish etc) and other exceptions and irregularities. I look forward to your feedback.
May the Type Design Gods Bless You Forever, Filip!!! ChrisL

This topic surfaces from time to time. It seems to usually be directed to us here at Adobe. There is usually some internal debate about it, so I don't want to imply that we (Adobe people) are all of one mind on the subject... but we've collectively considered and rejected such proposals before. For quite a while I was the person who was most directly confronted with glyph name issues, having to produce many of the fonts and create tools to handle them. (And we just revised the entire library, which gave me even more cause to agonize over glyph names.) Now that Miguel has joined us, he might be using and managing glyph names just as much -- and of course everyone else in the group also has some exposure and opinion. I dare say that for most of the OpenType era, but at least for the past five years or so, we have had fairly stable name standards at Adobe. But it's true that some names have changed. The thing is, we are usually pretty thoughtful about choosing names, and so the fact that we have changed them is the exact reason why I am reluctant to endorse some universal standard. No matter how thoughtful we are, something happens later to provoke a change to something. The more canonized a list becomes, the more trouble it becomes when a name changes -- and believe me, with a library of fonts as huge as Adobe's, that kind of trouble can be pretty unpleasant. We do have our own [[http://www.adobe.com/devnet/opentype/archives/glyph.html|glyph name conventions]], but I sense that this proposal is more about name suffixes, which are of no particular consequence in Adobe's recommendations. Some favor the feature-code-as-suffix approach (e.g. "seven.dnom"), while Adobe and others favor the descriptive approach (e.g. "seven.denominator"). For base names, Adobe name conventions have worked fairly well for us, as they rely on a manageable list of the most common names, falling back on reliable uni names for the rest. For suffixes, I am personally worried about "attaching" a layout feature to a glyph, when our association of features to glyphs is subject to change. There are already plenty of examples of Adobe fonts where we have moved a glyph from one feature to another. Of course now -- after the growing pains of OpenType in the early years -- we think we have it figured out pretty well, but there will be new issues and new features. It's already painful enough to consider moving a glyph from one feature to another because of compatibility issues, but things would get much scarier if I wanted to move, say, "A.salt" to the "ss01" feature. There could be cases of a glyph being accessed via two different features, or (as you illustrated) multiple alternates from one character in the same feature. All that can be handled one way or another with a name standard, but in can get complicated and increase the chances of mistakes (i.e. non-standard or conflicting names). A little freedom in this territory can be be nice to have. If you get the sense that I am fearful of change, you would be partially correct... but it's only because I've been trying for ten years to weed out all the peculiarities of our glyph names, and things are starting to finally settle down a bit. I like the names we use at Adobe, and don't mind that others use their own conventions. It seems like the current balance gives enough necessary structure and predictibility without boxing us in. By the way, regarding ligature names: They are pretty much settled in Adobe's name conventions. A name '"fi" refers to the single Unicode (ligature) character, whereas "f_i" refers to a ligature of two characters. It is indeed tricky to figure out suffixes for ligatures of alternates (e.g. "f_i.alt"), and even at Adobe I don't think we have much of a system other than semi-arbitrary "alt1", "alt2", and so on. So even if this thread doesn't result in some massive name overhaul, I think there's still a lot to discuss and improve in some areas.

2007-09-30 01:17:01 • k.l.
I don't think a standardised glyph naming scheme is necessary, beyond what Adobe define for glyph-to-character parsing, which is obviously a different need than that of font developers, hence the distinction between final glyph names in shipping fonts and what I call development or production glyph names. I'm also not convinced that there is a need for a standardised set of development glyph names, because as soon as you have two sets of names -- final and development -- that need to be convertible, you need a robust means of mapping from one set of glyphs to another and performing reversible conversions. And as soon as you have that, you have the means to switch between any other set of glyph naming conventions. Similarly, I don't think something like FontLab auto-generation of OTL lookups based on glyph names requires a standardised set of glyph names. What it does need is a means for developers to associate whatever suffixes they like with particular layout features. I don't want a tool like FontLab to force me into using particular glyph names: I want to be able to tell the tool what glyph names I was to use. Some things that we now do may be a bit simpler if there were a standard set of glyph names, but I'm not sure this justifies the effort to produce and agree on such a set. And what if, sometime after the set is agreed, I come up with a better method for, say, naming Devanagari glyphs or Arabic ligatures? Should I refrain from making use of that better method, for the sake of conformity? I know from looking back at the evolution of my own glyph naming systems that they have gradually improved as I work on different and different kinds of projects. So what I value is having powerful and flexible tools to allow me to convert between naming systems.

To “force all the important players in the font industry to accept it as a standard” has nasty repercussions, as John has already pointed out. Better to mirror the open-source method: write your standard, release it, get feedback, tweak it, and provide it as a good starting point. Users are then free to use it as a standard, or fork it off to suit their own needs and vision.
