Font Licensing on the Cloud

September 21, 2011 • faberdesign

I’m wondering if anyone has any insight about proper font licensing as it relates to cloud computing… Here’s a brief explanation:

I’m starting a business where we are creating custom journals (Bound Custom Journals - [[www.boundforanything.com]]). Some of the content that will go into a journal will be built on the fly on in an Amazon Web Services environment using ImageMagick. Those pages - usually calendar pages which get generated based on the date range you choose - need to be able to access a font file in order to properly render out those files, which then get populated into a pdf and sent to the printer.

Most font licenses allow for a user to install a font on up to 5 machines, however because we’re using AWS, during high traffic, that font may appear on more than 5 virtualized machines. During low traffic, it may not appear on any machines - the servers will be individually spun up and down based on demand (thus the beauty of cloud services).

My question is - Is a normal font license sufficient for this kind of use? Would love to hear anyone’s thoughts on this…

Thanks!

Cheers, Michael

Is a normal font license sufficient for this kind of use?

This is far beyond the limits of a normal font license. Most font licenses explicitly cover a number of simultaneous users equal to the number of licenses you purchased. Additionally, What you are doing is effectively embedding the font into software, which is prohibited by most retail font licenses. You need to contact the font designers directly to negotiate a license compliant with your system.

I wouldn’t bother with the “light EULA reading.” I believe that most retail EULAs are not written to address this situation (because this is not, strictly speaking, a retail type of situation — it’s more akin to OEM).

As Matthew Butterick suggested above, you’ll be better served by initiating a dialog directly with a representative of the foundry (or foundries) whose types you are interested in licensing and discussing your needs personally and in specifics. That way you can get a clearly tailored response. This kind of thing is just not going to be covered by a generic EULA.
